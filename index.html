HTML & CSS Reference

Notes on W3 HTML & CSS â€” elements, attributes, layout, and best practices for building websites.

Head Metadata

  • <meta charset="UTF-8"> â€” declares the character encoding. UTF-8 covers virtually all characters and should always be the first element inside <head>.
  • <meta name="viewport" content="width=device-width, initial-scale=1"> â€” required for responsive layouts on mobile. Without it, mobile browsers render at a desktop width and scale down.
  • <link rel="stylesheet" href="path/to/file.css"> â€” attaches an external CSS file. Place stylesheet links in <head> so the page doesn't flash unstyled content (FOUC).
  • <script src="file.js" defer></script> â€” the defer attribute delays script execution until HTML parsing is complete, preventing render blocking. Use async for scripts with no dependencies. Scripts without either attribute block parsing when encountered.
  • <title> â€” required element. Sets the browser tab label and is used by search engines as the page headline. Should be unique per page, ideally 50â€“60 characters.
  • <meta name="description" content="..."> â€” provides a summary shown as the snippet in search results. Recommended length is 150â€“160 characters.

Layout containers

  • <header> â€” introductory content for its nearest sectioning ancestor (page or section). Typically holds the site logo, branding, and primary navigation. There can be more than one per page as long as they're scoped to different sections.
  • <nav> â€” wraps a major block of navigation links. Use aria-label to distinguish multiple <nav> elements on the same page (e.g., aria-label="Primary", aria-label="Footer").
  • <main> â€” contains the dominant, unique content of the page. There must be only one <main> per page, and it should not be nested inside <article>, <aside>, <header>, <footer>, or <nav>.
  • <section> â€” a thematically grouped block that should have its own heading (h2â€“h6). Use <article> instead when the content could stand independently (e.g., a blog post). Use <aside> for tangentially related content like sidebars.
  • <footer> â€” closing information for its nearest sectioning ancestor. Typically holds copyright, secondary links, and contact info. Like <header>, multiple footers are valid when scoped to different sections.

Organizing CSS

  • Link external stylesheets in <head> with <link rel="stylesheet" href="...">. The browser downloads and applies them before rendering the page, preventing a flash of unstyled content.
  • For critical CSS (above-the-fold styles needed immediately), inline it inside a <style> tag in <head> to avoid a render-blocking network request. Keep this to a minimum.
  • Use <style> blocks for page-specific overrides. Avoid inline style="" attributes except for values that are genuinely dynamic or must be guaranteed to load with the element.
  • Load multiple stylesheets in dependency order â€” resets and base styles first, then component styles, then page-specific overrides.

Accessibility hooks

  • <html lang="en"> â€” the lang attribute is required for screen readers to select the correct voice/pronunciation engine, and for browser translation tools. Use a valid BCP 47 language tag (e.g., en, fr, pt-BR).
  • Every <section>, <nav>, <aside>, and <region> landmark should have an accessible name â€” either via a visible heading (h2â€“h6) or an aria-label/aria-labelledby attribute. Without one, screen readers announce the element type with no context.
  • Maintain heading hierarchy in document order: one <h1> per page, followed by <h2>s, then <h3>s, etc. Skipping levels (e.g., going from h2 to h4) breaks navigation for screen reader users who jump between headings.
  • All interactive elements must be keyboard-reachable and show a visible :focus indicator. Never remove the default outline without replacing it with an equally visible alternative.
  • Color alone should never be the only way information is conveyed â€” always pair color cues with text, icons, or patterns.

Text Content

  • Use exactly one <h1> per page â€” it names the page's primary topic. Follow with <h2> for major sections and <h3>â€“<h6> for subsections. Heading levels convey document structure to both screen readers and search engines; don't choose heading levels based on font size alone â€” use CSS for that.
  • <p> wraps paragraphs of text. Block-level element; cannot be nested inside itself or other block elements like <div> that are already inside a <p>.
  • <br> inserts a line break within inline content (e.g., inside a poem or address). Use sparingly â€” do not use it to create spacing between paragraphs; use margin on <p> elements instead.
  • <hr> represents a thematic break between paragraph-level content (e.g., a scene change in a story). It is a void element with no closing tag.
  • <strong> â€” marks text with strong importance. Renders as bold by default and carries semantic meaning (important to screen readers). Use font-weight: bold in CSS if you only want the visual effect.
  • <em> â€” marks text with stress emphasis. Renders as italic by default and affects how screen readers read the sentence aloud. Use font-style: italic in CSS for purely visual italics.
  • <mark> â€” highlights text as relevant in the current context (e.g., search result matches). Not the same as importance; use <strong> for importance.
  • <code> â€” inline code snippet, rendered in a monospace font. For multi-line code blocks, wrap <code> inside <pre> to preserve whitespace and line breaks.
  • <abbr title="..."> exposes the full expansion of an abbreviation on hover and to assistive technology. <cite> marks the title of a referenced work. <time datetime="YYYY-MM-DD"> encodes dates and times in a machine-readable format.
  • For readability, set line-height: 1.5â€“1.7 on body text, use generous paragraph margin-bottom, and keep line lengths between 60â€“80 characters using max-width on text containers.

Basic Lists

  • <ul> â€” unordered list. Use when the sequence of items does not matter. Renders with bullet points by default.
  1. <ol> â€” ordered list. Use when sequence matters (steps, rankings). Renders with numbers by default. Accepts start (e.g., start="3") to begin numbering from a value other than 1, and reversed to count down.
  • All list items must be wrapped in <li>. The only valid direct children of <ul> and <ol> are <li> elements â€” placing other elements directly inside the list is invalid HTML.
  • Lists can be nested by placing a new <ul> or <ol> inside an <li>. Style with list-style, padding, gap, and flexbox or grid on the <ul>/<ol> container.

Description List

  • <dl> â€” description list. Used for groups of termâ€“definition pairs such as glossaries, metadata, or FAQs. The only valid direct children are <dt> and <dd> (and optionally <div> to group pairs for styling).
  • <dt> â€” the term being defined. Multiple <dt> elements in a row are valid and indicate that the following <dd> applies to all of them.
  • <dd> â€” the description or value for the preceding <dt>. Multiple <dd> elements can follow a single <dt> for multiple definitions.

Links & nav

  • <a href="url"> â€” creates a hyperlink. href is required for the element to be interactive. Use relative paths for internal links (href="about.html") and absolute URLs for external ones.
  • For external links, always add target="_blank" rel="noopener noreferrer". target="_blank" opens in a new tab; noopener prevents the new page from accessing window.opener (a security risk); noreferrer also suppresses the HTTP referrer header.
  • title="..." adds an optional tooltip on hover. Don't rely on it for critical information â€” it's inaccessible to touch and keyboard users.
  • download â€” prompts the browser to download the linked file rather than navigate to it. Can optionally take a filename value: download="filename.pdf". Only works for same-origin URLs or data URIs.
  • Link text should describe the destination â€” avoid generic labels like "click here" or "read more," which are meaningless out of context for screen reader users.

Nav menus

  • Wrap navigation links in a <ul> inside <nav> â€” this provides list semantics so screen readers announce how many items are in the menu. Each link goes in an <li><a href="...">.
  • Add aria-label to <nav> when there are multiple nav regions on the page so they can be distinguished (e.g., aria-label="Primary", aria-label="Breadcrumb").
  • Mark the current page's link with aria-current="page" so screen readers identify it. An .active CSS class handles the visual style, but aria-current handles the semantic meaning.

States and Transitions

  • Add an .active class (or aria-current="page") to the nav link for the current page so users always know where they are.
  • :hover styles interactive elements on mouse-over. :focus applies when an element is focused via keyboard or programmatically. :focus-visible is preferred â€” it only shows the focus ring when the keyboard (not the mouse) is used, preventing the ring from appearing on mouse clicks while keeping it for keyboard navigation. Never set outline: none on :focus without a :focus-visible replacement.
  • Use transition: property duration easing; on the element's base state, not on the :hover rule, so the transition plays in both directions. Example: transition: background-color 0.2s ease;

Responsive Menus

  • On mobile, convert horizontal navigation into a vertical stack using a media query and flex-direction: column, or hide it behind a toggle button.
  • For a hamburger/toggle menu, use a <button> (not a <div> â€” buttons are keyboard-focusable and fire click events on Enter/Space by default). Toggle a class on the nav with JavaScript to show/hide it. Add aria-expanded="true/false" on the button to communicate the menu state to screen readers.
  • When the menu is hidden, use display: none or visibility: hidden rather than just moving it off-screen, so keyboard users cannot tab into hidden links.

Images and Media

  • <img src="url" alt="description"> â€” src and alt are both required. alt must describe the image's content or function. For decorative images that add no information, use an empty alt="" so screen readers skip it. Omitting alt entirely is invalid.
  • Set width and height attributes on <img> to match the image's natural dimensions â€” this lets the browser reserve space before the image loads, preventing layout shift. In CSS, use max-width: 100% and height: auto to keep it responsive within that reserved space.
  • <picture> with <source> elements allows serving different image files based on viewport size or format support. Each <source> uses a media attribute for breakpoints and srcset for the file path. The <img> inside <picture> is the required fallback and is where alt lives.
  • srcset on a plain <img> lets you provide multiple resolution variants (e.g., srcset="img-400.jpg 400w, img-800.jpg 800w") and the browser picks the best one based on device pixel ratio and viewport width.
  • <video> and <audio> support native browser playback. Common attributes: controls (shows playback UI â€” required for user accessibility), autoplay (use with muted, browsers block autoplay with sound), loop, preload. The poster attribute on <video> sets a thumbnail image shown before playback. Nest <source type="video/mp4"> elements inside for multiple formats, and include fallback text for browsers that don't support the element.
  • For YouTube/Vimeo embeds, wrap the <iframe> in a container with position: relative; padding-bottom: 56.25%; (56.25% = 9/16 for a 16:9 ratio), then set the iframe to position: absolute; top: 0; left: 0; width: 100%; height: 100%; to maintain aspect ratio at any width.
  • Add loading="lazy" to <img> elements below the fold to defer their network request until they're near the viewport. Do not use it on images above the fold â€” it delays images the user sees immediately.
  • Wrap images with captions in <figure> with a <figcaption> child. This semantically associates the caption with the image. <figure> can also wrap code blocks, charts, or other self-contained content.

Forms and Validation

  • <form action="url" method="post"> â€” action is where the form data is submitted; method is either GET (data appended to URL, suitable for searches) or POST (data in request body, required for sensitive data or large payloads).
  • Add autocomplete="on" (or field-specific values like autocomplete="email") to help browsers and password managers fill fields correctly. Add novalidate to the form element if you are handling validation entirely with JavaScript.
  • Every <input>, <select>, and <textarea> must have an associated <label>. Link them with a matching id on the input and for on the label: <label for="email"> / <input id="email">. Clicking the label then focuses the input. Do not rely solely on placeholder as a label â€” it disappears when the user types.
  • The required attribute prevents form submission if the field is empty and triggers the browser's built-in validation UI. Pair it with a :required CSS rule for visual indication.
  • Use aria-describedby="hint-id" on an input to associate it with a helper text element â€” screen readers will read the hint after the label. Use aria-invalid="true" when an input fails validation.
  • Group related inputs with <fieldset> and provide a group label with <legend> as its first child. This is especially important for radio buttons and checkboxes â€” without it, each option lacks context.
  • Style form controls consistently using the same CSS variables as the rest of your page. Use input:valid and input:invalid pseudo-classes to indicate validation state visually, but only after the user has interacted with the field (:user-invalid in modern browsers handles this automatically).
  • Security: always use HTTPS so form data is encrypted in transit. Never trust client-side validation alone â€” always validate and sanitize inputs on the server. Protect against CSRF with tokens, against XSS by escaping output, and against SQL injection by using parameterized queries.

Reuseable Components

  • Use meaningful, descriptive class names that reflect purpose rather than appearance â€” e.g., .btn-primary instead of .btn-red. This makes it easier to restyle by changing the CSS variable rather than renaming classes across the HTML.
  • Scope component styles to their own class so they don't bleed into unrelated elements. Avoid targeting bare element types (e.g., button { ... }) in component stylesheets â€” target the component class instead (.btn { ... }).
  • For buttons: write a base .btn class with shared properties (padding, border-radius, font, transition, cursor), then add modifier classes (.btn-primary, .btn-outline) that only override what differs. This avoids duplicating shared properties across variants.
  • Use CSS custom properties (--variable-name) to centralize design tokens (colors, spacing, border-radius). Component styles then reference the tokens, so a theme change in one place propagates everywhere.

Css Styling Essentials

  • CSS is a set of rules that control the visual presentation of HTML. Each rule has a selector that targets one or more elements, and a block of declarations that define propertyâ€“value pairs.
  • Each declaration follows the format property: value; â€” the semicolon is required between declarations. Example: border-radius: 24px;
  • The browser reads stylesheets top to bottom. When two rules target the same element with the same specificity, the one declared later wins. This is the "cascade" in Cascading Style Sheets.
  • Link an external stylesheet in <head> with <link rel="stylesheet" href="css/file.css">. Use a <style> block in <head> for small, page-specific styles. Use inline style="" attributes only for values that must be set at the element level (e.g., dynamic values from JavaScript).
  • There are four main selector types: type selectors (p), class selectors (.card), ID selectors (#header), and attribute selectors ([type="text"]). Classes are preferred for reusable styling; IDs should be used sparingly and mainly for anchor targets or JavaScript hooks.
  • Later rules override earlier ones only when the selectors have equal or higher specificity. A class (.btn-red) overrides a type rule (button) regardless of order.
  • Inline style="" attributes override any external or internal stylesheet declarations for the same property, short of !important.
  • Some CSS properties â€” like color, font-family, font-size, and line-height â€” are inherited from parent elements. Others, like padding, margin, border, and display, are not and must be set explicitly.
  • Only use !important as a last resort â€” it breaks the normal cascade and makes future debugging difficult. Prefer increasing selector specificity instead.
  • Specify colors as hex (#ff2627), RGB (rgb(255, 38, 39)), HSL (hsl(360, 100%, 57%)), or the modern oklch() format. HSL is often easier to adjust by eye â€” changing the L value lightens or darkens the color.
  • Control typography with font-family, font-size, font-weight (100â€“900, where 400 is normal and 700 is bold), line-height, letter-spacing, and text-align.
  • The box model: margin controls space outside the element's border, padding controls space between the content and the border, and border sits between them. Set box-sizing: border-box globally so width and height include padding and border.
  • The display property determines layout behavior. Key values: block (full-width, stacks vertically), inline (flows in text), inline-block (flows in text but respects width/height), flex (one-axis flexible layout), grid (two-axis grid layout).

Selectors & Specificity

  • The universal selector * matches every element. Commonly used in resets: *, *::before, *::after { box-sizing: border-box; }. High usage can have minor performance costs, but in practice this is rarely a real concern.
  • Selector types (in order of increasing specificity): type (p), class (.card), attribute ([href]), pseudo-class (:hover), ID (#main), inline style, !important.
  • Combinators: .card p â€” descendant (any p inside .card); .card > h3 â€” direct child only; h2 + p â€” adjacent sibling (immediately after); h2 ~ p â€” general sibling (any p after an h2 in the same parent).
  • Pseudo-classes target elements in a particular state: :hover, :focus, :focus-visible, :active, :visited, :checked, :disabled, :nth-child(n), :first-child, :last-child, :not(selector). Pseudo-elements target a part of an element: ::before, ::after, ::placeholder, ::first-line.
  • Specificity is calculated as a score of three parts: (ID count, class/attribute/pseudo-class count, element/pseudo-element count). For example, #nav .btn:hover scores (1,2,0). Higher score wins regardless of source order. Inline styles score (1,0,0,0) and beat everything except !important.
  • Write selectors from least to most specific within each stylesheet layer. Overly specific selectors are hard to override cleanly and make the codebase fragile.
  • Use browser DevTools (Inspect Element â†’ Styles panel) to see exactly which rules are applied, which are overridden (shown with strikethrough), and where each rule is declared.

    • Cascade & Inheritance

    • The cascade resolves conflicts between rules in three steps, in priority order: (1) Origin and importance â€” browser defaults lose to author styles; !important declarations win over normal ones from the same origin. (2) Specificity â€” higher specificity score wins. (3) Source order â€” when specificity is equal, the later declaration wins.
    • CSS layers (@layer) are a modern tool for managing cascade origin order explicitly, useful in large codebases to prevent specificity wars.
    • Inheritance: some properties like color, font-family, font-size, line-height, and visibility are automatically inherited from parent elements. Others like padding, margin, border, background, and display are not and start at their initial value.
    • Use inherit to force a non-inheriting property to take its parent's computed value. Use initial to reset a property to its CSS specification default. Use revert to reset to the browser's user-agent stylesheet value (e.g., making a <button> look like a browser default button again).
    • Specificity is calculated as (inline, ID, class/attribute/pseudo-class, element/pseudo-element). Example: #nav .btn:hover a = (0, 1, 2, 1).
    • !important overrides normal cascade for that property. Use it only to fix genuine conflicts you cannot resolve by adjusting specificity. Remove it once the underlying rule is corrected. Stacking !important declarations creates compounding cascade problems.
    • Use DevTools' Styles panel to trace exactly which rule is applying a value, why others are overridden, and what computed value the browser is using.

    HTML & CSS Typography

    • Typography controls how text is rendered â€” including font choice, size, weight, spacing, and alignment â€” and has a major effect on readability and perceived quality.
    • System font stacks use fonts already on the user's device, resulting in zero download cost and native-feeling text: font-family: system-ui, -apple-system, sans-serif;. Web-hosted fonts like Google Fonts give you specific typefaces but add a network request.
    • Set a base font on body with a fallback stack: font-family: 'Your Font', system-ui, sans-serif;. All elements inherit this unless overridden. Limit the page to 2â€“3 font families to keep the design cohesive.
    • Use relative units for font sizes: rem scales relative to the root (html) font size (typically 16px by default), so it respects the user's browser font preferences. em scales relative to the element's own parent â€” useful for components that should scale with their context, but can compound unexpectedly when nested.
    • font-weight accepts numeric values (100â€“900) or keywords (normal = 400, bold = 700). Only weights supported by the loaded font render correctly; unsupported weights are synthesized by the browser. font-style: italic renders text in italic.
    • line-height controls the vertical space between lines of text. A unitless value like 1.5 is preferred over px or em because it scales proportionally with the element's font size. letter-spacing adjusts space between characters.
    • text-align controls horizontal alignment (left, right, center, justify). text-transform changes case (uppercase, lowercase, capitalize). text-decoration adds or removes underlines; you can control its color and thickness separately with text-decoration-color and text-decoration-thickness.
    • Use clamp(min, preferred, max) to create fluid font sizes that scale with viewport width without media queries. Example: font-size: clamp(1rem, 2.5vw, 2rem);
    • Normalize type defaults on base elements â€” set body, h1â€“h6, p, and a explicitly rather than relying on browser defaults, which vary across browsers.
    • WCAG AA requires a minimum contrast ratio of 4.5:1 for normal text and 3:1 for large text (18pt+ or 14pt bold). Don't lock absolute font sizes â€” allow the user's browser zoom and font preference settings to work. Use semantic elements (<strong>, <em>) for meaning rather than just visual styling.

    Colors & Backgrounds

    • Color values can be specified as hex (#ff2627), rgb(255, 38, 39), hsl(360, 100%, 57%), or the modern oklch() format. HSL is often easiest to adjust â€” change L (lightness) to create tints and shades. oklch() is perceptually uniform, meaning equal changes in L always look equally different.
    • Define colors as CSS custom properties (variables) at the :root level: --primary-red: #ff2627;. Reference them with var(--primary-red). This makes global color changes a one-line edit and enables theming.
    • background-color fills the element with a solid color. background-image sets an image or gradient. Both can be layered by separating values with commas (images render on top). background-size: cover scales the image to fill the container; background-position: center centers it; background-repeat: no-repeat prevents tiling.
    • CSS gradients: linear-gradient(direction, color1, color2) transitions colors in a straight line; radial-gradient(shape, color1, color2) radiates outward from a center point; conic-gradient(color1, color2) sweeps around a center point like a pie chart. Gradients are image values, so they go in background-image, not background-color.
    • To overlay a gradient on a background image, comma-separate them in background-image: background-image: linear-gradient(...), url(image.jpg); â€” the gradient is rendered on top.
    • Data URIs and CSS gradients can create subtle textures and patterns without additional image files, keeping asset requests low.
    • Ensure all text meets WCAG AA contrast requirements: 4.5:1 for normal text, 3:1 for large text (18pt+). Use a contrast checker tool to verify color pairs before shipping.

    Box Model

    • Every HTML element is rendered as a rectangular box composed of four layers from inside out: content â†’ padding â†’ border â†’ margin.
    • Content is the area where text and child elements appear. Padding is transparent space between the content and the border. Border wraps the padding and content. Margin is transparent space outside the border that separates the element from its neighbors.
    • By default, width and height apply only to the content area. With box-sizing: border-box;, they instead apply to the total box including padding and border. Apply this globally: *, *::before, *::after { box-sizing: border-box; } â€” otherwise width calculations are unintuitive.
    • DevTools' box model diagram (Inspect Element â†’ Computed tab) shows the exact pixel values for each layer of the selected element.
    • Margin collapsing: when two block elements are adjacent vertically, their margins collapse to the larger of the two values rather than adding together. A parent element and its first or last child can also share margins unless the parent has padding, a border, or triggers a block formatting context.
    • To prevent unexpected margin collapse between a parent and its first child, add padding-top: 1px or a border to the parent, or give the parent overflow: hidden or display: flow-root.
    • Shorthand: margin: 24px 16px; sets 24px top/bottom and 16px left/right. Four values follow clockwise: top, right, bottom, left (TRBL). The same pattern applies to padding and border-width.
    • Define a spacing scale using custom properties (e.g., --space-sm: 8px; --space-md: 16px; --space-lg: 24px;) and apply them consistently. This keeps margins and padding proportional across the page and makes adjustments easy.

    Layout Fundamentals

    • Normal flow: block-level elements (<div>, <p>, <section>) stack vertically and take the full available width. Inline elements (<span>, <a>, <strong>) flow horizontally with text and ignore width/height. Inline-block elements flow like inline but respect width and height.
    • Change an element's layout behavior with display: value;. Common values: block, inline, inline-block, flex, inline-flex, grid, inline-grid, none (removes from layout entirely).
    • float: left or float: right removes an element from normal flow and allows text to wrap around it. The parent container collapses in height and doesn't contain floated children. Clear a float container with overflow: auto, overflow: hidden, or the clearfix method (::after { content: ""; display: table; clear: both; }).
    • Prefer flexbox for one-dimensional layouts and grid for two-dimensional layouts. Floats are mostly legacy â€” you'll encounter them in older codebases.
    • Elements render in DOM (source) order by default. Positioning and z-index can change the visual stacking order independently of source order, but source order still determines tab order for keyboard users.
    • A new stacking context is formed by elements with: position + a z-index other than auto; opacity less than 1; transform; filter; isolation: isolate; and several others. Elements inside a stacking context are z-ordered relative to each other, not to the rest of the page. This is a common source of z-index issues.

    Positioning & Z-index

    • Positioning lets you place elements independently of normal document flow. Use it for overlays, tooltips, badges, sticky headers, and fixed UI chrome â€” not as a general layout tool (use flexbox/grid for that).
    • Position values: static (default, ignores top/left/right/bottom and z-index); relative (stays in normal flow but offsets from its original position using top/left/right/bottom; creates a containing block for absolutely positioned children); absolute (removed from flow, positioned relative to the nearest non-static ancestor); fixed (removed from flow, positioned relative to the viewport and stays there during scroll); sticky (acts as relative until it reaches a scroll threshold, then behaves as fixed within its scroll container).
    • Combine position with top, right, bottom, left, and inset (shorthand for all four) to control placement. These offsets have no effect on static elements.
    • A common pattern: set position: relative on a parent, then position: absolute; top: 0; right: 0; on a child (like a badge or close icon) to anchor it to the parent's corner.
    • z-index only works on elements with a position value other than static, and on flex or grid children. Higher z-index renders on top. Elements in different stacking contexts are compared at the context level, not element by element â€” this is why a high z-index sometimes appears behind a lower-z-index element.
    • For position: sticky to work: the sticky element must have a top, left, right, or bottom threshold set, and its scroll container must have overflow set to something other than visible (or no overflow set at all). A common bug is a parent with overflow: hidden breaking sticky behavior.
    • Use DevTools to inspect stacking contexts when overlays misbehave â€” it highlights which element creates each context.

    Flexbox Layout

    • Flexbox is a one-dimensional layout model â€” it arranges items along either a row or a column and handles distribution of space and alignment along those axes. Ideal for navigation bars, card rows, form layouts, and centering.
    • Enable flexbox on a container with display: flex; (block-level) or display: inline-flex; (inline-level). The immediate children become flex items automatically.
    • flex-direction: row (default) lays items left-to-right; row-reverse reverses that. column stacks vertically; column-reverse reverses it. This sets the main axis.
    • The main axis follows flex-direction. Control spacing along it with justify-content: flex-start, flex-end, center, space-between, space-around, space-evenly. The cross axis is perpendicular â€” align items along it with align-items on the container or align-self on individual items.
    • Use gap: row-gap column-gap; for consistent spacing between items. This is preferred over margins because it only adds space between items, not on the outer edges.
    • flex shorthand on items: flex: grow shrink basis;. flex-grow defines how much of remaining space the item takes (0 = don't grow). flex-shrink defines how the item shrinks when space is tight (0 = don't shrink). flex-basis sets the starting size before space is distributed. Common shorthands: flex: 1; (equal width, same as 1 1 0%), flex: 0 0 300px; (fixed 300px, doesn't grow or shrink).
    • flex-wrap: wrap; lets items wrap onto new rows or columns when they exceed the container width. Without it, items shrink to fit or overflow. align-content aligns the rows/columns when wrapping occurs.
    • order changes the visual rendering order of flex items without altering DOM order. Note: DOM order still controls tab order, so changing visual order with order can create confusing keyboard navigation if overused.

    Grid Layout

    • CSS Grid is a two-dimensional layout system â€” it handles rows and columns simultaneously. Best for page-level layouts, dashboards, image galleries, and any structure that needs alignment in both axes.
    • Enable with display: grid; (block-level) or display: inline-grid;. Immediate children become grid items.
    • Define columns with grid-template-columns and rows with grid-template-rows. Use px for fixed tracks, fr (fractional unit) to distribute remaining space, auto to size to content, or minmax(min, max) for flexible bounds.
    • gap (or row-gap / column-gap) sets track spacing. The old grid-gap property is deprecated; use gap.
    • Fixed-width first column, equal remaining: grid-template-columns: 300px 1fr 1fr;. The 1fr units share the space left after the 300px column.
    • Responsive fluid grid without media queries: grid-template-columns: repeat(auto-fit, minmax(220px, 1fr)); â€” auto-fit collapses empty tracks so existing items stretch to fill; auto-fill keeps empty tracks. minmax(220px, 1fr) sets a minimum column width of 220px and a maximum of one fractional unit.
    • Grid items auto-place in source order by default. Explicitly place items with grid-column: start / end; and grid-row: start / end; using line numbers. Use span to span multiple tracks: grid-column: 1 / span 3;
    • Named areas: define a visual layout in ASCII with grid-template-areas on the container, then assign items to areas with grid-area: name;. Use . for empty cells.
    • Alignment inside tracks: justify-items (inline axis) and align-items (block axis) on the container. Override per item with justify-self and align-self. Use place-items: center; as shorthand for both axes.
    • Alignment of the whole grid within its container: justify-content and align-content, or place-content shorthand. Useful when the total track size is smaller than the container.
    • For responsive breakpoints, adjust grid-template-columns and grid-template-areas inside media queries to reflow the layout at different screen widths.

    Layout Patterns

    • Use flexbox and grid as your primary layout tools. Avoid floats for new layouts â€” they exist in legacy code but are not the right tool for modern page structure.
    • Media + text pair: use a two-column grid (grid-template-columns: 1fr 1fr;) that collapses to a single column on small screens with a media query. The media element (image/video) occupies one column and text the other.
    • App shell pattern: a fixed outer grid or flexbox defines the sidebar, header, and main content area. The header and sidebar stay constant across page navigation; only the main area content changes. Use grid-template-areas to name the regions and reassign them in media queries for mobile.
    • Card grid: use display: grid; grid-template-columns: repeat(auto-fit, minmax(260px, 1fr)); for a responsive card layout that automatically adjusts column count without media queries.
    • Centering: display: grid; place-items: center; or display: flex; justify-content: center; align-items: center; on a container are the cleanest ways to center a child both horizontally and vertically.

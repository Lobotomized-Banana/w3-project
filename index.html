Head Metadata

  • <meta charset="UTF-8"> — declares the character encoding. UTF-8 covers virtually all characters and should always be the first element inside <head>.
  • <meta name="viewport" content="width=device-width, initial-scale=1"> — required for responsive layouts on mobile. Without it, mobile browsers render at a desktop width and scale down.
  • <link rel="stylesheet" href="path/to/file.css"> — attaches an external CSS file. Place stylesheet links in <head> so the page doesn't flash unstyled content (FOUC).
  • <script src="file.js" defer></script> — the defer attribute delays script execution until HTML parsing is complete, preventing render blocking. Use async for scripts with no dependencies. Scripts without either attribute block parsing when encountered.
  • <title> — required element. Sets the browser tab label and is used by search engines as the page headline. Should be unique per page, ideally 50–60 characters.
  • <meta name="description" content="..."> — provides a summary shown as the snippet in search results. Recommended length is 150–160 characters.

Layout containers

  • <header> — introductory content for its nearest sectioning ancestor (page or section). Typically holds the site logo, branding, and primary navigation. There can be more than one per page as long as they're scoped to different sections.
  • <nav> — wraps a major block of navigation links. Use aria-label to distinguish multiple <nav> elements on the same page (e.g., aria-label="Primary", aria-label="Footer").
  • <main> — contains the dominant, unique content of the page. There must be only one <main> per page, and it should not be nested inside <article>, <aside>, <header>, <footer>, or <nav>.
  • <section> — a thematically grouped block that should have its own heading (h2–h6). Use <article> instead when the content could stand independently (e.g., a blog post). Use <aside> for tangentially related content like sidebars.
  • <footer> — closing information for its nearest sectioning ancestor. Typically holds copyright, secondary links, and contact info. Like <header>, multiple footers are valid when scoped to different sections.

Organizing CSS

  • Link external stylesheets in <head> with <link rel="stylesheet" href="...">. The browser downloads and applies them before rendering the page, preventing a flash of unstyled content.
  • For critical CSS (above-the-fold styles needed immediately), inline it inside a <style> tag in <head> to avoid a render-blocking network request. Keep this to a minimum.
  • Use <style> blocks for page-specific overrides. Avoid inline style="" attributes except for values that are genuinely dynamic or must be guaranteed to load with the element.
  • Load multiple stylesheets in dependency order — resets and base styles first, then component styles, then page-specific overrides.

Accessibility hooks

  • <html lang="en"> — the lang attribute is required for screen readers to select the correct voice/pronunciation engine, and for browser translation tools. Use a valid BCP 47 language tag (e.g., en, fr, pt-BR).
  • Every <section>, <nav>, <aside>, and <region> landmark should have an accessible name — either via a visible heading (h2–h6) or an aria-label/aria-labelledby attribute. Without one, screen readers announce the element type with no context.
  • Maintain heading hierarchy in document order: one <h1> per page, followed by <h2>s, then <h3>s, etc. Skipping levels (e.g., going from h2 to h4) breaks navigation for screen reader users who jump between headings.
  • All interactive elements must be keyboard-reachable and show a visible :focus indicator. Never remove the default outline without replacing it with an equally visible alternative.
  • Color alone should never be the only way information is conveyed — always pair color cues with text, icons, or patterns.

Text Content

  • Use exactly one <h1> per page — it names the page's primary topic. Follow with <h2> for major sections and <h3>–<h6> for subsections. Heading levels convey document structure to both screen readers and search engines; don't choose heading levels based on font size alone — use CSS for that.
  • <p> wraps paragraphs of text. Block-level element; cannot be nested inside itself or other block elements like <div> that are already inside a <p>.
  • <br> inserts a line break within inline content (e.g., inside a poem or address). Use sparingly — do not use it to create spacing between paragraphs; use margin on <p> elements instead.
  • <hr> represents a thematic break between paragraph-level content (e.g., a scene change in a story). It is a void element with no closing tag.
  • <strong> — marks text with strong importance. Renders as bold by default and carries semantic meaning (important to screen readers). Use font-weight: bold in CSS if you only want the visual effect.
  • <em> — marks text with stress emphasis. Renders as italic by default and affects how screen readers read the sentence aloud. Use font-style: italic in CSS for purely visual italics.
  • <mark> — highlights text as relevant in the current context (e.g., search result matches). Not the same as importance; use <strong> for importance.
  • <code> — inline code snippet, rendered in a monospace font. For multi-line code blocks, wrap <code> inside <pre> to preserve whitespace and line breaks.
  • <abbr title="..."> exposes the full expansion of an abbreviation on hover and to assistive technology. <cite> marks the title of a referenced work. <time datetime="YYYY-MM-DD"> encodes dates and times in a machine-readable format.
  • For readability, set line-height: 1.5–1.7 on body text, use generous paragraph margin-bottom, and keep line lengths between 60–80 characters using max-width on text containers.

Basic Lists

  • <ul> — unordered list. Use when the sequence of items does not matter. Renders with bullet points by default.
  1. <ol> — ordered list. Use when sequence matters (steps, rankings). Renders with numbers by default. Accepts start (e.g., start="3") to begin numbering from a value other than 1, and reversed to count down.
  • All list items must be wrapped in <li>. The only valid direct children of <ul> and <ol> are <li> elements — placing other elements directly inside the list is invalid HTML.
  • Lists can be nested by placing a new <ul> or <ol> inside an <li>. Style with list-style, padding, gap, and flexbox or grid on the <ul>/<ol> container.

Description List

  • <dl> — description list. Used for groups of term–definition pairs such as glossaries, metadata, or FAQs. The only valid direct children are <dt> and <dd> (and optionally <div> to group pairs for styling).
  • <dt> — the term being defined. Multiple <dt> elements in a row are valid and indicate that the following <dd> applies to all of them.
  • <dd> — the description or value for the preceding <dt>. Multiple <dd> elements can follow a single <dt> for multiple definitions.

Links & nav

  • <a href="url"> — creates a hyperlink. href is required for the element to be interactive. Use relative paths for internal links (href="about.html") and absolute URLs for external ones.
  • For external links, always add target="_blank" rel="noopener noreferrer". target="_blank" opens in a new tab; noopener prevents the new page from accessing window.opener (a security risk); noreferrer also suppresses the HTTP referrer header.
  • title="..." adds an optional tooltip on hover. Don't rely on it for critical information — it's inaccessible to touch and keyboard users.
  • download — prompts the browser to download the linked file rather than navigate to it. Can optionally take a filename value: download="filename.pdf". Only works for same-origin URLs or data URIs.
  • Link text should describe the destination — avoid generic labels like "click here" or "read more," which are meaningless out of context for screen reader users.

Nav menus

  • Wrap navigation links in a <ul> inside <nav> — this provides list semantics so screen readers announce how many items are in the menu. Each link goes in an <li><a href="...">.
  • Add aria-label to <nav> when there are multiple nav regions on the page so they can be distinguished (e.g., aria-label="Primary", aria-label="Breadcrumb").
  • Mark the current page's link with aria-current="page" so screen readers identify it. An .active CSS class handles the visual style, but aria-current handles the semantic meaning.

States and Transitions

  • Add an .active class (or aria-current="page") to the nav link for the current page so users always know where they are.
  • :hover styles interactive elements on mouse-over. :focus applies when an element is focused via keyboard or programmatically. :focus-visible is preferred — it only shows the focus ring when the keyboard (not the mouse) is used, preventing the ring from appearing on mouse clicks while keeping it for keyboard navigation. Never set outline: none on :focus without a :focus-visible replacement.
  • Use transition: property duration easing; on the element's base state, not on the :hover rule, so the transition plays in both directions. Example: transition: background-color 0.2s ease;

Responsive Menus

  • On mobile, convert horizontal navigation into a vertical stack using a media query and flex-direction: column, or hide it behind a toggle button.
  • For a hamburger/toggle menu, use a <button> (not a <div> — buttons are keyboard-focusable and fire click events on Enter/Space by default). Toggle a class on the nav with JavaScript to show/hide it. Add aria-expanded="true/false" on the button to communicate the menu state to screen readers.
  • When the menu is hidden, use display: none or visibility: hidden rather than just moving it off-screen, so keyboard users cannot tab into hidden links.

Images and Media

  • <img src="url" alt="description"> — src and alt are both required. alt must describe the image's content or function. For decorative images that add no information, use an empty alt="" so screen readers skip it. Omitting alt entirely is invalid.
  • Set width and height attributes on <img> to match the image's natural dimensions — this lets the browser reserve space before the image loads, preventing layout shift. In CSS, use max-width: 100% and height: auto to keep it responsive within that reserved space.
  • <picture> with <source> elements allows serving different image files based on viewport size or format support. Each <source> uses a media attribute for breakpoints and srcset for the file path. The <img> inside <picture> is the required fallback and is where alt lives.
  • srcset on a plain <img> lets you provide multiple resolution variants (e.g., srcset="img-400.jpg 400w, img-800.jpg 800w") and the browser picks the best one based on device pixel ratio and viewport width.
  • <video> and <audio> support native browser playback. Common attributes: controls (shows playback UI — required for user accessibility), autoplay (use with muted, browsers block autoplay with sound), loop, preload. The poster attribute on <video> sets a thumbnail image shown before playback. Nest <source type="video/mp4"> elements inside for multiple formats, and include fallback text for browsers that don't support the element.
  • For YouTube/Vimeo embeds, wrap the <iframe> in a container with position: relative; padding-bottom: 56.25%; (56.25% = 9/16 for a 16:9 ratio), then set the iframe to position: absolute; top: 0; left: 0; width: 100%; height: 100%; to maintain aspect ratio at any width.
  • Add loading="lazy" to <img> elements below the fold to defer their network request until they're near the viewport. Do not use it on images above the fold — it delays images the user sees immediately.
  • Wrap images with captions in <figure> with a <figcaption> child. This semantically associates the caption with the image. <figure> can also wrap code blocks, charts, or other self-contained content.

Forms and Validation

  • <form action="url" method="post"> — action is where the form data is submitted; method is either GET (data appended to URL, suitable for searches) or POST (data in request body, required for sensitive data or large payloads).
  • Add autocomplete="on" (or field-specific values like autocomplete="email") to help browsers and password managers fill fields correctly. Add novalidate to the form element if you are handling validation entirely with JavaScript.
  • Every <input>, <select>, and <textarea> must have an associated <label>. Link them with a matching id on the input and for on the label: <label for="email"> / <input id="email">. Clicking the label then focuses the input. Do not rely solely on placeholder as a label — it disappears when the user types.
  • The required attribute prevents form submission if the field is empty and triggers the browser's built-in validation UI. Pair it with a :required CSS rule for visual indication.
  • Use aria-describedby="hint-id" on an input to associate it with a helper text element — screen readers will read the hint after the label. Use aria-invalid="true" when an input fails validation.
  • Group related inputs with <fieldset> and provide a group label with <legend> as its first child. This is especially important for radio buttons and checkboxes — without it, each option lacks context.
  • Style form controls consistently using the same CSS variables as the rest of your page. Use input:valid and input:invalid pseudo-classes to indicate validation state visually, but only after the user has interacted with the field (:user-invalid in modern browsers handles this automatically).
  • Security: always use HTTPS so form data is encrypted in transit. Never trust client-side validation alone — always validate and sanitize inputs on the server. Protect against CSRF with tokens, against XSS by escaping output, and against SQL injection by using parameterized queries.

Reuseable Components

  • Use meaningful, descriptive class names that reflect purpose rather than appearance — e.g., .btn-primary instead of .btn-red. This makes it easier to restyle by changing the CSS variable rather than renaming classes across the HTML.
  • Scope component styles to their own class so they don't bleed into unrelated elements. Avoid targeting bare element types (e.g., button { ... }) in component stylesheets — target the component class instead (.btn { ... }).
  • For buttons: write a base .btn class with shared properties (padding, border-radius, font, transition, cursor), then add modifier classes (.btn-primary, .btn-outline) that only override what differs. This avoids duplicating shared properties across variants.
  • Use CSS custom properties (--variable-name) to centralize design tokens (colors, spacing, border-radius). Component styles then reference the tokens, so a theme change in one place propagates everywhere.

Css Styling Essentials

  • CSS is a set of rules that control the visual presentation of HTML. Each rule has a selector that targets one or more elements, and a block of declarations that define property–value pairs.
  • Each declaration follows the format property: value; — the semicolon is required between declarations. Example: border-radius: 24px;
  • The browser reads stylesheets top to bottom. When two rules target the same element with the same specificity, the one declared later wins. This is the "cascade" in Cascading Style Sheets.
  • Link an external stylesheet in <head> with <link rel="stylesheet" href="css/file.css">. Use a <style> block in <head> for small, page-specific styles. Use inline style="" attributes only for values that must be set at the element level (e.g., dynamic values from JavaScript).
  • There are four main selector types: type selectors (p), class selectors (.card), ID selectors (#header), and attribute selectors ([type="text"]). Classes are preferred for reusable styling; IDs should be used sparingly and mainly for anchor targets or JavaScript hooks.
  • Later rules override earlier ones only when the selectors have equal or higher specificity. A class (.btn-red) overrides a type rule (button) regardless of order.
  • Inline style="" attributes override any external or internal stylesheet declarations for the same property, short of !important.
  • Some CSS properties — like color, font-family, font-size, and line-height — are inherited from parent elements. Others, like padding, margin, border, and display, are not and must be set explicitly.
  • Only use !important as a last resort — it breaks the normal cascade and makes future debugging difficult. Prefer increasing selector specificity instead.
  • Specify colors as hex (#ff2627), RGB (rgb(255, 38, 39)), HSL (hsl(360, 100%, 57%)), or the modern oklch() format. HSL is often easier to adjust by eye — changing the L value lightens or darkens the color.
  • Control typography with font-family, font-size, font-weight (100–900, where 400 is normal and 700 is bold), line-height, letter-spacing, and text-align.
  • The box model: margin controls space outside the element's border, padding controls space between the content and the border, and border sits between them. Set box-sizing: border-box globally so width and height include padding and border.
  • The display property determines layout behavior. Key values: block (full-width, stacks vertically), inline (flows in text), inline-block (flows in text but respects width/height), flex (one-axis flexible layout), grid (two-axis grid layout).

Selectors & Specificity

  • The universal selector * matches every element. Commonly used in resets: *, *::before, *::after { box-sizing: border-box; }. High usage can have minor performance costs, but in practice this is rarely a real concern.
  • Selector types (in order of increasing specificity): type (p), class (.card), attribute ([href]), pseudo-class (:hover), ID (#main), inline style, !important.
  • Combinators: .card p — descendant (any p inside .card); .card > h3 — direct child only; h2 + p — adjacent sibling (immediately after); h2 ~ p — general sibling (any p after an h2 in the same parent).
  • Pseudo-classes target elements in a particular state: :hover, :focus, :focus-visible, :active, :visited, :checked, :disabled, :nth-child(n), :first-child, :last-child, :not(selector). Pseudo-elements target a part of an element: ::before, ::after, ::placeholder, ::first-line.
  • Specificity is calculated as a score of three parts: (ID count, class/attribute/pseudo-class count, element/pseudo-element count). For example, #nav .btn:hover scores (1,2,0). Higher score wins regardless of source order. Inline styles score (1,0,0,0) and beat everything except !important.
  • Write selectors from least to most specific within each stylesheet layer. Overly specific selectors are hard to override cleanly and make the codebase fragile.
  • Use browser DevTools (Inspect Element → Styles panel) to see exactly which rules are applied, which are overridden (shown with strikethrough), and where each rule is declared.

    • Cascade & Inheritance

    • The cascade resolves conflicts between rules in three steps, in priority order: (1) Origin and importance — browser defaults lose to author styles; !important declarations win over normal ones from the same origin. (2) Specificity — higher specificity score wins. (3) Source order — when specificity is equal, the later declaration wins.
    • CSS layers (@layer) are a modern tool for managing cascade origin order explicitly, useful in large codebases to prevent specificity wars.
    • Inheritance: some properties like color, font-family, font-size, line-height, and visibility are automatically inherited from parent elements. Others like padding, margin, border, background, and display are not and start at their initial value.
    • Use inherit to force a non-inheriting property to take its parent's computed value. Use initial to reset a property to its CSS specification default. Use revert to reset to the browser's user-agent stylesheet value (e.g., making a <button> look like a browser default button again).
    • Specificity is calculated as (inline, ID, class/attribute/pseudo-class, element/pseudo-element). Example: #nav .btn:hover a = (0, 1, 2, 1).
    • !important overrides normal cascade for that property. Use it only to fix genuine conflicts you cannot resolve by adjusting specificity. Remove it once the underlying rule is corrected. Stacking !important declarations creates compounding cascade problems.
    • Use DevTools' Styles panel to trace exactly which rule is applying a value, why others are overridden, and what computed value the browser is using.

    HTML & CSS Typography

    • Typography controls how text is rendered — including font choice, size, weight, spacing, and alignment — and has a major effect on readability and perceived quality.
    • System font stacks use fonts already on the user's device, resulting in zero download cost and native-feeling text: font-family: system-ui, -apple-system, sans-serif;. Web-hosted fonts like Google Fonts give you specific typefaces but add a network request.
    • Set a base font on body with a fallback stack: font-family: 'Your Font', system-ui, sans-serif;. All elements inherit this unless overridden. Limit the page to 2–3 font families to keep the design cohesive.
    • Use relative units for font sizes: rem scales relative to the root (html) font size (typically 16px by default), so it respects the user's browser font preferences. em scales relative to the element's own parent — useful for components that should scale with their context, but can compound unexpectedly when nested.
    • font-weight accepts numeric values (100–900) or keywords (normal = 400, bold = 700). Only weights supported by the loaded font render correctly; unsupported weights are synthesized by the browser. font-style: italic renders text in italic.
    • line-height controls the vertical space between lines of text. A unitless value like 1.5 is preferred over px or em because it scales proportionally with the element's font size. letter-spacing adjusts space between characters.
    • text-align controls horizontal alignment (left, right, center, justify). text-transform changes case (uppercase, lowercase, capitalize). text-decoration adds or removes underlines; you can control its color and thickness separately with text-decoration-color and text-decoration-thickness.
    • Use clamp(min, preferred, max) to create fluid font sizes that scale with viewport width without media queries. Example: font-size: clamp(1rem, 2.5vw, 2rem);
    • Normalize type defaults on base elements — set body, h1–h6, p, and a explicitly rather than relying on browser defaults, which vary across browsers.
    • WCAG AA requires a minimum contrast ratio of 4.5:1 for normal text and 3:1 for large text (18pt+ or 14pt bold). Don't lock absolute font sizes — allow the user's browser zoom and font preference settings to work. Use semantic elements (<strong>, <em>) for meaning rather than just visual styling.

    Colors & Backgrounds

    • Color values can be specified as hex (#ff2627), rgb(255, 38, 39), hsl(360, 100%, 57%), or the modern oklch() format. HSL is often easiest to adjust — change L (lightness) to create tints and shades. oklch() is perceptually uniform, meaning equal changes in L always look equally different.
    • Define colors as CSS custom properties (variables) at the :root level: --primary-red: #ff2627;. Reference them with var(--primary-red). This makes global color changes a one-line edit and enables theming.
    • background-color fills the element with a solid color. background-image sets an image or gradient. Both can be layered by separating values with commas (images render on top). background-size: cover scales the image to fill the container; background-position: center centers it; background-repeat: no-repeat prevents tiling.
    • CSS gradients: linear-gradient(direction, color1, color2) transitions colors in a straight line; radial-gradient(shape, color1, color2) radiates outward from a center point; conic-gradient(color1, color2) sweeps around a center point like a pie chart. Gradients are image values, so they go in background-image, not background-color.
    • To overlay a gradient on a background image, comma-separate them in background-image: background-image: linear-gradient(...), url(image.jpg); — the gradient is rendered on top.
    • Data URIs and CSS gradients can create subtle textures and patterns without additional image files, keeping asset requests low.
    • Ensure all text meets WCAG AA contrast requirements: 4.5:1 for normal text, 3:1 for large text (18pt+). Use a contrast checker tool to verify color pairs before shipping.

    Box Model

    • Every HTML element is rendered as a rectangular box composed of four layers from inside out: content → padding → border → margin.
    • Content is the area where text and child elements appear. Padding is transparent space between the content and the border. Border wraps the padding and content. Margin is transparent space outside the border that separates the element from its neighbors.
    • By default, width and height apply only to the content area. With box-sizing: border-box;, they instead apply to the total box including padding and border. Apply this globally: *, *::before, *::after { box-sizing: border-box; } — otherwise width calculations are unintuitive.
    • DevTools' box model diagram (Inspect Element → Computed tab) shows the exact pixel values for each layer of the selected element.
    • Margin collapsing: when two block elements are adjacent vertically, their margins collapse to the larger of the two values rather than adding together. A parent element and its first or last child can also share margins unless the parent has padding, a border, or triggers a block formatting context.
    • To prevent unexpected margin collapse between a parent and its first child, add padding-top: 1px or a border to the parent, or give the parent overflow: hidden or display: flow-root.
    • Shorthand: margin: 24px 16px; sets 24px top/bottom and 16px left/right. Four values follow clockwise: top, right, bottom, left (TRBL). The same pattern applies to padding and border-width.
    • Define a spacing scale using custom properties (e.g., --space-sm: 8px; --space-md: 16px; --space-lg: 24px;) and apply them consistently. This keeps margins and padding proportional across the page and makes adjustments easy.

    Layout Fundamentals

    • Normal flow: block-level elements (<div>, <p>, <section>) stack vertically and take the full available width. Inline elements (<span>, <a>, <strong>) flow horizontally with text and ignore width/height. Inline-block elements flow like inline but respect width and height.
    • Change an element's layout behavior with display: value;. Common values: block, inline, inline-block, flex, inline-flex, grid, inline-grid, none (removes from layout entirely).
    • float: left or float: right removes an element from normal flow and allows text to wrap around it. The parent container collapses in height and doesn't contain floated children. Clear a float container with overflow: auto, overflow: hidden, or the clearfix method (::after { content: ""; display: table; clear: both; }).
    • Prefer flexbox for one-dimensional layouts and grid for two-dimensional layouts. Floats are mostly legacy — you'll encounter them in older codebases.
    • Elements render in DOM (source) order by default. Positioning and z-index can change the visual stacking order independently of source order, but source order still determines tab order for keyboard users.
    • A new stacking context is formed by elements with: position + a z-index other than auto; opacity less than 1; transform; filter; isolation: isolate; and several others. Elements inside a stacking context are z-ordered relative to each other, not to the rest of the page. This is a common source of z-index issues.

    Positioning & Z-index

    • Positioning lets you place elements independently of normal document flow. Use it for overlays, tooltips, badges, sticky headers, and fixed UI chrome — not as a general layout tool (use flexbox/grid for that).
    • Position values: static (default, ignores top/left/right/bottom and z-index); relative (stays in normal flow but offsets from its original position using top/left/right/bottom; creates a containing block for absolutely positioned children); absolute (removed from flow, positioned relative to the nearest non-static ancestor); fixed (removed from flow, positioned relative to the viewport and stays there during scroll); sticky (acts as relative until it reaches a scroll threshold, then behaves as fixed within its scroll container).
    • Combine position with top, right, bottom, left, and inset (shorthand for all four) to control placement. These offsets have no effect on static elements.
    • A common pattern: set position: relative on a parent, then position: absolute; top: 0; right: 0; on a child (like a badge or close icon) to anchor it to the parent's corner.
    • z-index only works on elements with a position value other than static, and on flex or grid children. Higher z-index renders on top. Elements in different stacking contexts are compared at the context level, not element by element — this is why a high z-index sometimes appears behind a lower-z-index element.
    • For position: sticky to work: the sticky element must have a top, left, right, or bottom threshold set, and its scroll container must have overflow set to something other than visible (or no overflow set at all). A common bug is a parent with overflow: hidden breaking sticky behavior.
    • Use DevTools to inspect stacking contexts when overlays misbehave — it highlights which element creates each context.

    Flexbox Layout

    • Flexbox is a one-dimensional layout model — it arranges items along either a row or a column and handles distribution of space and alignment along those axes. Ideal for navigation bars, card rows, form layouts, and centering.
    • Enable flexbox on a container with display: flex; (block-level) or display: inline-flex; (inline-level). The immediate children become flex items automatically.
    • flex-direction: row (default) lays items left-to-right; row-reverse reverses that. column stacks vertically; column-reverse reverses it. This sets the main axis.
    • The main axis follows flex-direction. Control spacing along it with justify-content: flex-start, flex-end, center, space-between, space-around, space-evenly. The cross axis is perpendicular — align items along it with align-items on the container or align-self on individual items.
    • Use gap: row-gap column-gap; for consistent spacing between items. This is preferred over margins because it only adds space between items, not on the outer edges.
    • flex shorthand on items: flex: grow shrink basis;. flex-grow defines how much of remaining space the item takes (0 = don't grow). flex-shrink defines how the item shrinks when space is tight (0 = don't shrink). flex-basis sets the starting size before space is distributed. Common shorthands: flex: 1; (equal width, same as 1 1 0%), flex: 0 0 300px; (fixed 300px, doesn't grow or shrink).
    • flex-wrap: wrap; lets items wrap onto new rows or columns when they exceed the container width. Without it, items shrink to fit or overflow. align-content aligns the rows/columns when wrapping occurs.
    • order changes the visual rendering order of flex items without altering DOM order. Note: DOM order still controls tab order, so changing visual order with order can create confusing keyboard navigation if overused.

    Grid Layout

    • CSS Grid is a two-dimensional layout system — it handles rows and columns simultaneously. Best for page-level layouts, dashboards, image galleries, and any structure that needs alignment in both axes.
    • Enable with display: grid; (block-level) or display: inline-grid;. Immediate children become grid items.
    • Define columns with grid-template-columns and rows with grid-template-rows. Use px for fixed tracks, fr (fractional unit) to distribute remaining space, auto to size to content, or minmax(min, max) for flexible bounds.
    • gap (or row-gap / column-gap) sets track spacing. The old grid-gap property is deprecated; use gap.
    • Fixed-width first column, equal remaining: grid-template-columns: 300px 1fr 1fr;. The 1fr units share the space left after the 300px column.
    • Responsive fluid grid without media queries: grid-template-columns: repeat(auto-fit, minmax(220px, 1fr)); — auto-fit collapses empty tracks so existing items stretch to fill; auto-fill keeps empty tracks. minmax(220px, 1fr) sets a minimum column width of 220px and a maximum of one fractional unit.
    • Grid items auto-place in source order by default. Explicitly place items with grid-column: start / end; and grid-row: start / end; using line numbers. Use span to span multiple tracks: grid-column: 1 / span 3;
    • Named areas: define a visual layout in ASCII with grid-template-areas on the container, then assign items to areas with grid-area: name;. Use . for empty cells.
    • Alignment inside tracks: justify-items (inline axis) and align-items (block axis) on the container. Override per item with justify-self and align-self. Use place-items: center; as shorthand for both axes.
    • Alignment of the whole grid within its container: justify-content and align-content, or place-content shorthand. Useful when the total track size is smaller than the container.
    • For responsive breakpoints, adjust grid-template-columns and grid-template-areas inside media queries to reflow the layout at different screen widths.

    Responsive Design

    Responsive design ensures every layout looks and works great on any device. Combine flexible grid systems, fluid typography, and media queries to adapt content gracefully.

    Viewport Basics

    • Always include the viewport meta tag: <meta name="viewport" content="width=device-width, initial-scale=1">. Without it, mobile browsers render at a desktop width and scale everything down, breaking your layout.
    • Design mobile-first — write your base styles for narrow screens, then layer on media queries to enhance for tablets and desktops. This approach keeps CSS leaner and forces you to prioritise essential content.
    • Use relative units (rem, vw, %) rather than px wherever possible so elements scale naturally with the viewport and user font-size preferences.

    Fluid Grids

    • Use Flexbox and Grid with fractional units (fr) to create layouts that flow and resize rather than overflow. Fractional units divide up leftover space evenly between tracks.
    • Apply max-width to text containers (e.g. max-width: 65ch;) to prevent line lengths from becoming unreadably wide on large screens.
    • Use repeat(auto-fit, minmax(min, 1fr)) on grid containers for automatic, breakpoint-free column counts — columns form when there's room and collapse to one when there isn't.

    Responsive Typography

    • Use clamp(min, preferred, max) for font sizes: e.g. font-size: clamp(1rem, 2vw, 1.5rem);. The browser picks a value between the minimum and maximum, keeping text readable at every viewport width.
    • Scale headings, line heights, and spacing together so the vertical rhythm holds at all sizes. If the heading grows, the surrounding white space should grow proportionally.
    • Consider defining fluid spacing tokens with CSS custom properties — e.g. --space-md: clamp(1rem, 3vw, 2rem); — so you can update spacing site-wide from one place.

    Media Queries

    • A breakpoint is the screen width at which your layout needs to change. Choose breakpoints based on where the design breaks, not around specific device sizes, since device sizes change constantly.
    • Write mobile-first: base styles apply to all screens; @media (min-width: ...) queries progressively enhance for larger viewports. This is typically more maintainable than overriding desktop styles downward.
    • Common starting points: min-width: 480px (large phones), min-width: 768px (tablets), min-width: 1200px (wide desktops) — but always adjust to suit your content.
    :root {
  --space: clamp(1rem, 2vw, 1.5rem);

  gap: var(--space);
}
@media (min-width: 768px) {

@media (min-width: 1200px) {

}

    Container Queries

    • Standard media queries react to the viewport width; container queries (@container) react to the width of a parent element. This makes components truly self-contained and reusable in different layout contexts.
    • Opt a parent into container query tracking with container-type: inline-size;. Then use @container (min-width: 400px) { ... } inside child selectors.
    • Browser support is now good (all modern browsers), but still provide sensible default styles as a fallback for older environments.

    Responsive Media

    • Make all images fluid by default: img { max-width: 100%; height: auto; }. This prevents images from overflowing their containers while maintaining their aspect ratio.
    • Use the <picture> element with <source media="..."> attributes to serve different image files at different breakpoints — useful for art direction (e.g. a cropped portrait on mobile, a wide landscape on desktop).
    • For embedded videos and iframes, preserve aspect ratio with aspect-ratio: 16 / 9; combined with width: 100%;, or use the older padding-percentage hack for broader support.
    • Use object-fit: cover; on <img> elements inside fixed-size containers to crop cleanly without distorting the image.

    Testing Strategy

    • Use browser DevTools device mode to quickly preview common breakpoints and simulate touch events. Resize the viewport manually to catch edge cases between presets.
    • Test on real devices when possible — DevTools emulation doesn't fully replicate touch target size, font rendering, or scrolling behaviour on actual hardware.
    • Check landscape and portrait orientations, and test with the browser's minimum and maximum font-size settings to ensure your fluid type still holds.
    • Audit performance: responsive sites should serve appropriately sized images and avoid loading desktop-weight resources on mobile. Use the Network tab in DevTools to check payload sizes.

    Data Tables

    Wide tables overflow on small screens. Rather than forcing horizontal scrolling, consider restructuring the data for mobile.

    • The simplest approach is overflow-x: auto; on a wrapper element — this contains the overflow and lets the user scroll the table horizontally without the rest of the page scrolling.
    • For a richer experience, use display: block; on tr, td, and th at narrow widths to transform rows into stacked card-like blocks. Use data-label attributes and CSS ::before pseudo-elements to re-inject column headings next to each value.

    Responsive Forms

    Forms need to be easy to complete on both a desktop keyboard and a phone touchscreen.

    • Set width: 100%; on inputs and buttons at narrow viewports so they become easy tap targets that fill the available width.
    • Use the correct type attribute on inputs (type="email", type="tel", type="number") so mobile browsers display the appropriate keyboard automatically.

    Micro-interactions & Touch Considerations

    Small animations and state changes make interfaces feel responsive to input — but they must work across pointer, keyboard, and touch.

    • Never style only :hover. Always pair it with :focus-visible so keyboard users get the same visual feedback. On touch devices, :hover can be sticky or entirely absent.
    • Wrap animations in @media (prefers-reduced-motion: no-preference) { ... } so they only run for users who haven't requested reduced motion at the OS level. Users with vestibular disorders can experience nausea from excessive motion.
    • All interactive elements — buttons, links, form controls — should have a minimum tap target size of 44×44 px as recommended by WCAG. Use min-height and padding rather than setting a fixed height, so the element can still grow with its content.
    • Use CSS transition on transform and opacity for animations rather than animating width, height, or margin — transform and opacity changes are handled by the compositor thread and don't cause layout recalculations, so they stay smooth even on low-powered devices.

    CSS Custom Properties (Variables)

    • CSS custom properties — commonly called variables — are defined with a -- prefix and read back with the var() function. They are case-sensitive: --Color-Brand and --color-brand are distinct properties.
    • Declare global tokens on :root, which targets the <html> element and gives them the highest possible scope. Any element in the document can then inherit and consume them via var(--token-name).
    • Custom properties are part of the cascade and inherit just like color or font-size. Redefining a variable on a container element — e.g. .card { --surface: #1e293b; } — scopes the new value to that element and all of its descendants, leaving the global definition untouched.
    • Provide a fallback value as the second argument to var(): var(--space-md, 1rem). The fallback is used whenever the referenced variable has not been defined in the current scope, acting as a safe default rather than causing an invalid value.
    • Variables work seamlessly inside calc(), clamp(), and other CSS functions: padding: calc(var(--space-md) * 1.5); or font-size: clamp(var(--text-min), 3vw, var(--text-max));. The variable is substituted first, then the function evaluates.
    • Switch themes at runtime by toggling a data-theme attribute on <html> and redefining the same token names under that selector. Because all components reference the same variable names, every colour, spacing value, and shadow updates automatically with a single attribute change — no JavaScript needs to touch individual elements.
    • The @media (prefers-color-scheme: dark) query can redefine custom properties directly, giving users an automatic dark theme that respects their OS preference without any JavaScript at all.
    • JavaScript can read a variable with getComputedStyle(el).getPropertyValue('--token-name').trim() and write one with el.style.setProperty('--token-name', value). Writing to document.documentElement updates the global scope; writing to a specific element scopes the change to that element only.
    • Use @supports (--custom: property) to detect custom property support before applying variable-dependent rules. All modern browsers support custom properties; this guard is only necessary when targeting very old browsers such as IE 11, which has no support at all.

    CSS Functions & Calculations

    CSS functions let you compute values, blend colors, and respond to context without extra JavaScript. Combine them with custom properties to build smart, responsive design systems.

    Math Helpers

    • calc() — performs arithmetic across mixed units directly in CSS. Example: width: calc(100% - 2rem); subtracts a fixed rem value from a fluid percentage — something that is impossible with a plain static value. Spaces around the operator are required by the spec.
    • min() / max() — pick the smallest or largest value from a comma-separated list. Example: width: min(600px, 100%); keeps an element at most 600px wide but never overflows its container. Multiple values and mixed units are supported.
    • clamp(min, preferred, max) — constrains a value to a responsive range. The middle (preferred) value is typically a viewport unit so it scales fluidly; the first and third arguments act as hard floor and ceiling. Example: font-size: clamp(1rem, 2.5vw, 1.5rem); scales with the viewport but never smaller than 1rem or larger than 1.5rem.
    /* Fluid spacing scale using clamp() — scales between breakpoints
   without any media queries. Custom properties make the scale
   reusable across the entire project. */
:root {
  --space-xs: clamp(0.5rem,  1vw,   0.75rem);
  --space-sm: clamp(0.75rem, 1.5vw, 1.125rem);
  --space-lg: clamp(1.5rem,  3vw,   2.5rem);
}

.card {
  padding:       var(--space-lg);
  margin-block:  var(--space-sm);
  /* calc() uses the custom property as an operand */
  border-radius: calc(var(--space-xs) * 2);
}

/* width can never exceed 800px, but is otherwise fluid */
.content-area {
  width: min(800px, 100% - var(--space-lg) * 2);
}

    CSS Transitions

    CSS transitions animate changes to CSS properties, creating smooth hover states, focus styles, and UI feedback without JavaScript. They activate automatically whenever a targeted property changes value — no scripting required.

    Core Properties

    • transition-property — names the CSS property (or properties) to animate, e.g. background-color, transform, opacity. Use all to target every animatable property at once, but avoid it in production: it causes the browser to watch every property for changes, which is wasteful and can produce unintended animations on properties you never meant to transition.
    • transition-duration — total running time of the animation, written as a time value (200ms, 0.3s). There is no default — a missing or zero duration means the change is instant.
    • transition-timing-function — the easing curve that controls acceleration through the animation. Common keywords are ease (default — slow start, fast middle, slow end), linear, ease-in (starts slow), ease-out (ends slow), and ease-in-out. Custom curves use cubic-bezier().
    • transition-delay — how long the browser waits before starting the animation after the property change is detected. Useful for staggering a sequence of elements.
    • transition shorthand — sets all four in one declaration in the order property duration timing-function delay. Example: transition: background 180ms ease, transform 180ms ease;. Multiple transitions are comma-separated, each targeting a different property.
    /* Shorthand: property | duration | timing-function | delay
   Two properties transitioned independently in one rule. */
.cta {
  display: inline-flex;
  align-items: center;
  gap: 12px;
  padding: 12px 22px;
  border-radius: 999px;
  background: #2563eb;
  color: #fff;
  text-decoration: none;
  /* background and transform each get their own curve */
  transition: background 180ms ease, transform 180ms ease;
}

/* :focus-visible ensures keyboard users see the same feedback
   as pointer users — never style :hover alone */
.cta:hover,
.cta:focus-visible {
  background: #1d4ed8;   /* darker blue */
  transform: translateY(-2px); /* subtle lift */
}

    Timing Functions

    • ease — the default; accelerates quickly then decelerates. Works well for most UI interactions because it feels natural to the human eye.
    • linear — constant speed throughout. Best for looping animations (spinners, progress bars) where you want no perceived acceleration.
    • ease-in — starts slow, ends fast. Use sparingly for elements leaving the screen (exits), where the acceleration feels intentional.
    • ease-out — starts fast, ends slow. Ideal for elements entering the screen (entrances) — the deceleration implies the element has arrived and settled.
    • ease-in-out — slow at both ends, fast in the middle. Good for longer transitions where you want a smooth, polished feel throughout.
    • cubic-bezier(x1, y1, x2, y2) — define any custom curve using two control points. Tools like cubic-bezier.com let you preview curves visually before committing them to code. Values can exceed 0–1 on the Y axis to create overshoot (bounce) effects.
    • steps(n, direction) — divides the transition into n equal discrete jumps instead of a smooth curve. Use it for sprite-sheet animations, typewriter effects, or any scenario where intermediate states should not be visible.
    /* Custom spring-like overshoot using cubic-bezier.
   Y values above 1.0 cause the element to "bounce" past
   the target before settling — impossible with keyword easings. */
.spring {
  transition: transform 400ms cubic-bezier(0.34, 1.56, 0.64, 1);
}
.spring:hover {
  transform: scale(1.08);
}

/* steps() — jumps between frames, no interpolation.
   Perfect for a typewriter cursor blink. */
.cursor {
  animation: blink 1s steps(1, end) infinite;
}
@keyframes blink {
  50% { opacity: 0; }
}

    Responsive Motion & Accessibility

    Hover states and motion effects do not exist on touch screens or for users who have requested reduced motion at the operating system level. Any interaction that relies solely on :hover or on animation must have a respectful fallback.

    • Wrap all non-essential motion inside @media (prefers-reduced-motion: no-preference) { ... } so transitions only run when the user has not requested reduced motion. The inverse query (prefers-reduced-motion: reduce) can be used to explicitly supply a stripped-down fallback with no movement. This is an operating-system accessibility setting; honoring it prevents discomfort or harm for users with vestibular disorders.
    • Keep interactive feedback durations between 150ms and 300ms. Shorter feels snappy and responsive; beyond 300ms starts to feel sluggish. Reserve longer durations (400–600ms) only for complex transitions like page-level layout changes.
    • Avoid large transform distances or significant layout-shifting transitions — a button that moves 80px on hover is jarring; 2–4px is the practical sweet spot for a lift effect.
    • Always pair :hover styles with :focus-visible so keyboard users receive identical visual feedback. :focus-visible (rather than :focus) shows the indicator only when navigating by keyboard, avoiding the focus ring on mouse clicks.
    • Do not transition width, height, top, or left — these trigger full layout recalculations on every frame. Prefer transform: scale() and transform: translate(), which are composited by the GPU and do not affect document flow.
    /* Safe default: no motion at all */
.btn {
  background: #2563eb;
  color: #fff;
}
.btn:hover,
.btn:focus-visible {
  background: #1d4ed8;
}

/* Motion layer: only applied when the user has NOT
   requested reduced motion at the OS level */
@media (prefers-reduced-motion: no-preference) {
  .btn {
    transition: background 180ms ease, transform 180ms ease;
  }
  .btn:hover,
  .btn:focus-visible {
    transform: translateY(-2px);
  }
}

/* GPU-friendly: use transform instead of top/left/width/height */
.panel {
  /* BAD  — triggers layout: width: 200px → width: 400px  */
  /* GOOD — composited, no reflow:                         */
  transform: scaleX(0.5);
  transform-origin: left center;
  transition: transform 250ms ease-out;
}
.panel.open {
  transform: scaleX(1);
}

    2D Transforms

    • translate(x, y) — moves an element along the X and/or Y axis without disturbing surrounding elements. Positive X moves right; positive Y moves down. Single-axis shorthand: translateX() / translateY(). Accepts any CSS length unit or percentage; percentages are relative to the element's own dimensions, making translate(-50%, -50%) a reliable centring technique.
    • scale(x, y) — resizes an element around its transform origin. A value of 1 is unchanged, 2 doubles the size, 0.5 halves it. If only one argument is given, both axes scale equally. Use scaleX() or scaleY() to stretch or compress on a single axis.
    • rotate(angle) — spins the element clockwise for positive values, counter-clockwise for negative. Accepts deg, rad, grad, or turn units. Example: rotate(0.25turn) and rotate(90deg) are equivalent.
    • skew(x-angle, y-angle) — slants the element along each axis. Use sparingly — skew distorts text and can reduce readability. Single-axis shorthand: skewX() / skewY().
    • Multiple transforms can be chained in a single transform declaration, separated by spaces: transform: translateY(-4px) scale(1.02) rotate(2deg);. The browser applies them right-to-left, so order matters — rotate then translate produces a different result than translate then rotate.
    /* Chain multiple 2D transforms — order is right-to-left.
   This scales first, then lifts the already-scaled element. */
.card:hover,
.card:focus-visible {
  transform: translateY(-6px) scale(1.02);
  transition: transform 220ms ease;
}

/* Percentage translate: centres absolutely-positioned elements
   regardless of their dimensions */
.modal {
  position: absolute;
  top: 50%;
  left: 50%;
  transform: translate(-50%, -50%); /* pulls back by half own width/height */
}

    3D Transforms

    • perspective(value) — applied directly on the transform property of the element being transformed (inline perspective). Smaller values produce a more dramatic depth effect; larger values flatten it. Example: transform: perspective(700px) rotateY(20deg);. Alternatively, set perspective: 800px; on the parent container so all child elements share the same vanishing point, which creates a more cohesive scene.
    • rotateX(angle) / rotateY(angle) / rotateZ(angle) — rotate around each 3D axis. rotateX tips the top toward or away from the viewer; rotateY swings left and right; rotateZ is equivalent to the 2D rotate().
    • translateZ(value) — moves the element toward or away from the viewer along the Z axis. Positive values bring it closer; negative values push it back. Requires perspective to be visible.
    • transform-style: preserve-3d; — set on a parent to ensure its children are positioned in 3D space rather than flattened into the parent's plane. Without it, nested 3D transforms have no effect on descendants.
    • backface-visibility: hidden; — hides the reverse face of an element when it is rotated more than 90°. Essential for card-flip effects to prevent the back face showing through while the front is visible.
    /* 3D card tilt on hover — perspective set inline so each card
   has its own vanishing point */
.grid {
  display: grid;
  gap: 24px;
  grid-template-columns: repeat(auto-fit, minmax(220px, 1fr));
  padding: clamp(24px, 6vw, 80px);
}

.card {
  padding: 32px;
  border-radius: 24px;
  background: linear-gradient(145deg, #1e293b, #0f172a);
  color: #fff;
  /* baseline: flat, no tilt */
  transform: perspective(700px) rotateX(0deg) rotateY(0deg);
  transition: transform 220ms ease, box-shadow 220ms ease;
}

.card:hover,
.card:focus-visible {
  transform: perspective(700px) rotateX(6deg) rotateY(-6deg) translateY(-6px);
  box-shadow: 0 24px 40px rgb(15 23 42 / 0.35);
}

/* Card-flip pattern — requires preserve-3d on the container */
.flip-container {
  perspective: 1000px;
}
.flip-card {
  transform-style: preserve-3d;
  transition: transform 500ms ease;
}
.flip-card.flipped {
  transform: rotateY(180deg);
}
.flip-card .front,
.flip-card .back {
  backface-visibility: hidden;
  position: absolute;
  inset: 0;
}
.flip-card .back {
  transform: rotateY(180deg);
}

    Transform Origin

    • By default, every transform is applied relative to the element's centre (50% 50%). transform-origin shifts that anchor point, changing how rotation, scaling, and skewing behave.
    • Accepts keyword pairs (top left, bottom right), length values (0 0), or percentages (100% 0%). A third value sets the Z-axis origin for 3D transforms.
    • Common patterns: transform-origin: top center; for a dropdown unfolding from its trigger; transform-origin: left center; for a sliding drawer that grows rightward from its attachment point; transform-origin: bottom center; for a tooltip that scales up from its pointer.
    /* Dropdown menu expands downward from the trigger edge,
   not from its own centre */
.dropdown {
  transform-origin: top center;
  transform: scaleY(0);
  transition: transform 200ms ease-out;
}
.dropdown.open {
  transform: scaleY(1);
}

/* Tooltip scales from its tail point at the bottom */
.tooltip {
  transform-origin: bottom center;
  transform: scale(0);
  transition: transform 150ms cubic-bezier(0.34, 1.56, 0.64, 1);
}
.tooltip.visible {
  transform: scale(1);
}

    Performance Tips

    • Transforms are GPU-accelerated — the browser hands them to the compositor thread and animates them without triggering layout or paint recalculations. This makes transform: translate() dramatically cheaper than changing top, left, margin, or width.
    • will-change: transform; is a hint to the browser that this element is about to be transformed, allowing it to promote the element to its own compositor layer in advance. Use it sparingly and only when you have measured a real performance problem — overusing it consumes GPU memory and can actually slow pages down.
    • Animating opacity alongside transform is safe; both are composited properties. Avoid animating filter, clip-path, or border-radius simultaneously with transforms on low-powered devices — these may fall back to software rendering.
    • Always provide a @media (prefers-reduced-motion: reduce) fallback that removes or simplifies large transform movements. A 180° flip or a 100px translateY that is dramatic on desktop can cause discomfort for users with vestibular disorders.
    /* Default: instant state change, no motion */
.panel {
  transform: translateX(-100%);
}
.panel.open {
  transform: translateX(0);
}

/* Motion enhancement — only added when the user hasn't
   requested reduced motion at the OS level */
@media (prefers-reduced-motion: no-preference) {
  .panel {
    transition: transform 300ms ease-out;
    /* hint to compositor — only add if you've measured a
       real jank problem; remove after profiling */
    will-change: transform;
  }
}

    CSS Animations

    CSS animations use @keyframes rules to define intermediate states along a timeline, then apply them to elements via animation-* properties. Unlike transitions, animations run automatically — no state change is required to trigger them.

    @keyframes

    • @keyframes name { … } — the at-rule that defines the animation sequence. The name you give it is what you reference via animation-name. Inside the rule, keyframe selectors mark points in the timeline where specific CSS values should be applied.
    • Use the keyword from for the start state (equivalent to 0%) and to for the end state (equivalent to 100%). Percentages let you add intermediate steps — e.g. 50% or 0%, 100% as a comma-separated selector list to apply the same styles at multiple points.
    • If you omit from (0%) or to (100%), the browser uses the element's current computed style as the missing start or end state. This is useful for animating from an element's natural position without repeating those values inside the keyframe.
    • Properties marked !important inside a keyframe are ignored — per the MDN specification, !important declarations in @keyframes rules have no effect and the property falls back to its cascaded value.
    • If multiple @keyframes rules share the same name, the last one parsed by the browser wins. Rules do not cascade or merge — only one keyframe set is ever used for a given name.
    /* Named keyframe sequence — referenced by animation-name */
@keyframes slide-in {
  from {
    transform: translateX(-100%);
    opacity: 0;
  }
  to {
    transform: translateX(0);
    opacity: 1;
  }
}

/* Multi-stop sequence using percentages */
@keyframes pulse {
  0%   { transform: scale(1);    box-shadow: 0 0 0 0 rgba(37, 99, 235, 0.4); }
  70%  { transform: scale(1.05); box-shadow: 0 0 0 10px rgba(37, 99, 235, 0); }
  100% { transform: scale(1);    box-shadow: 0 0 0 0 rgba(37, 99, 235, 0); }
}

    Animation Properties

    • animation-name — references the @keyframes rule to apply. Multiple names can be comma-separated to run several animations simultaneously on the same element. Use none to disable an animation without removing other animation declarations in the cascade.
    • animation-duration — the total time for one full cycle, in seconds (s) or milliseconds (ms). There is no default — if this is missing or set to 0s the animation fires instantly. The unit is required even for zero: 0s not 0.
    • animation-timing-function — the easing curve applied within each keyframe interval, not across the whole animation. Accepts the same keywords as transition-timing-function: ease (default), linear, ease-in, ease-out, ease-in-out, cubic-bezier(), and steps().
    • animation-delay — how long to wait before the first iteration begins. A negative value is valid and causes the animation to start mid-cycle — useful for staggering items in a list so they feel offset without actually delaying each one.
    • animation-iteration-count — how many times the animation repeats. Use a positive number or the keyword infinite for a looping animation. Fractional values (e.g. 0.5) run a partial cycle before stopping.
    • animation-direction — controls the playback direction each iteration: normal (always forward, default), reverse (always backward), alternate (forward then backward, reversing each cycle — useful for looping animations that need to return to their start), alternate-reverse (backward then forward).
    • animation-fill-mode — what styles apply outside the active animation window. none (default) applies nothing before or after; forwards keeps the final keyframe's styles after the animation ends; backwards applies the first keyframe's styles during the delay period; both applies both behaviours simultaneously.
    • animation-play-state — running (default) or paused. Toggle this with JavaScript or a :hover rule to pause and resume an animation mid-cycle without losing its current position.
    • animation shorthand — sets all sub-properties in order: duration | easing | delay | iteration-count | direction | fill-mode | play-state | name. Multiple animations are comma-separated. When both animation-duration and animation-delay are time values in the shorthand, the first time value is always duration and the second is delay.
    /* Shorthand: duration | easing | delay | iteration | direction | fill-mode | name */
.badge {
  animation: 2s ease-in-out 0s infinite alternate both pulse;
}

/* The same written out longhand */
.badge {
  animation-name:            pulse;
  animation-duration:        2s;
  animation-timing-function: ease-in-out;
  animation-delay:           0s;
  animation-iteration-count: infinite;
  animation-direction:       alternate;
  animation-fill-mode:       both;
  animation-play-state:      running;
}

/* Pause on hover via play-state — animation resumes from
   the same position when the cursor leaves */
.badge:hover {
  animation-play-state: paused;
}

/* Stagger three items using a negative delay — each starts
   mid-cycle so they feel offset without actually waiting */
.item:nth-child(1) { animation: 1.2s ease infinite slide-in; }
.item:nth-child(2) { animation: 1.2s ease -0.4s infinite slide-in; }
.item:nth-child(3) { animation: 1.2s ease -0.8s infinite slide-in; }

    animation-fill-mode in Depth

    • forwards is the most commonly needed value — without it, a one-shot animation (e.g. a card sliding in on page load) snaps back to its original position as soon as it finishes. forwards locks the element at the final keyframe.
    • backwards is only noticeable when there is an animation-delay. Without it the element shows its base CSS styles during the delay; with backwards, it immediately assumes the first keyframe's styles, which prevents a visual jump when the animation finally starts.
    • For entrance animations — fade in, slide in — the combination animation-fill-mode: both; animation-iteration-count: 1; is the most reliable pattern. The element starts at the first keyframe and ends locked at the last, regardless of what the base CSS says.

    Accessibility & Performance

    • Always wrap non-essential motion inside @media (prefers-reduced-motion: no-preference) { … }. This is the same requirement covered in the Transitions section — CSS animations are subject to the same rule. Users who have enabled the reduced-motion setting at the OS level (often for vestibular disorders) should receive instant state changes, not animated ones.
    • Animate only transform and opacity for smooth, GPU-composited animations. Animating width, height, top, left, margin, or padding triggers layout recalculation on every frame, which is expensive and causes jank on lower-powered devices.
    • will-change: transform; can pre-promote an element to its own GPU layer before the animation begins, reducing the initial frame drop. Use it sparingly — only when profiling confirms a performance problem — because each promoted layer consumes GPU memory. Remove it after the animation completes if possible.
    • Prefer animation-iteration-count: infinite over JavaScript-driven loop timers for looping effects. Infinite CSS animations are entirely off the main thread once running, while JS setInterval competes with other tasks and can stutter.
    /* Safe base: no animation at all. The element is already in its
   final position — no FOUC, no layout shift. */
.hero-text {
  opacity: 1;
  transform: translateY(0);
}

/* Motion layer: only applied when the user has NOT
   requested reduced motion at the OS level */
@media (prefers-reduced-motion: no-preference) {
  .hero-text {
    opacity: 0;
    animation: fade-up 600ms ease-out forwards;
  }
}

@keyframes fade-up {
  from {
    opacity: 0;
    transform: translateY(24px);
  }
  to {
    opacity: 1;
    transform: translateY(0);
  }
}

/* GPU-friendly looping spinner — transform and opacity only */
.spinner {
  animation: spin 800ms linear infinite;
}
@keyframes spin {
  to { transform: rotate(360deg); }
}

    CSS Effects & Filters

    CSS visual effects let you add depth, atmosphere, and image manipulation entirely in the browser — no JavaScript, no image editing software. This covers shadows, opacity, filter functions, backdrop filters, and blend modes.

    box-shadow

    • box-shadow applies one or more drop shadows to an element's box. The value syntax is: offset-x offset-y blur-radius spread-radius color. The blur and spread radii are optional — omitting both produces a sharp, solid shadow.
    • offset-x and offset-y control horizontal and vertical displacement. Negative offset-x places the shadow to the left; negative offset-y places it above the element. Setting both to 0 centers the shadow behind the element, useful for a pure glow effect when combined with blur.
    • blur-radius — the larger the value, the bigger and softer the shadow. Negative values are not allowed. A value of 0 produces a sharp edge.
    • spread-radius — positive values expand the shadow beyond the element's dimensions; negative values shrink it. This is useful for creating an inner-glow effect at small negative values.
    • Add the inset keyword before the offsets to flip the shadow from an outer drop-shadow to an inner one — the shadow appears inside the element's border, above the background but below the content. Per MDN, inset shadows are drawn inside the border even if the border is transparent.
    • Multiple shadows are comma-separated and rendered front-to-back (first in the list renders on top). This is how layered, multi-colour glow effects are built.
    • Unlike filter: drop-shadow(), box-shadow always follows the element's rectangular bounding box — it does not trace the actual shape of partially transparent content. Use filter: drop-shadow() instead if you need the shadow to follow irregular shapes like PNG cut-outs or SVGs.
    /* offset-x | offset-y | blur | spread | color */
.card {
  box-shadow: 0 4px 12px 0 rgb(0 0 0 / 0.15);
}

/* inset keyword — shadow appears inside the box */
.inset-well {
  box-shadow: inset 0 2px 6px rgb(0 0 0 / 0.3);
}

/* Multiple shadows — first renders on top.
   Outer ambient shadow + tighter directional shadow layered. */
.elevated {
  box-shadow:
    0 1px 2px  rgb(0 0 0 / 0.10),
    0 8px 24px rgb(0 0 0 / 0.18);
}

/* Glow effect — both offsets at 0, spread expands the blur */
.glow {
  box-shadow: 0 0 0 4px rgb(37 99 235 / 0.4),
              0 0 24px 0 rgb(37 99 235 / 0.3);
}

/* Animating box-shadow is expensive (triggers paint).
   Prefer animating opacity on a ::after pseudo-element instead. */
.card {
  position: relative;
}
.card::after {
  content: "";
  position: absolute;
  inset: 0;
  border-radius: inherit;
  box-shadow: 0 16px 40px rgb(0 0 0 / 0.3);
  opacity: 0;
  transition: opacity 200ms ease;
}
.card:hover::after {
  opacity: 1; /* only opacity changes — no paint recalculation */
}

    text-shadow

    • text-shadow applies one or more shadows to the text content of an element. The syntax is the same as box-shadow minus the spread radius and the inset keyword — both are unsupported on text shadows. Values are: offset-x offset-y blur-radius color.
    • Like box-shadow, multiple shadows are comma-separated and stacked front-to-back. Layering a very tight, slightly-offset dark shadow under a softer wide shadow produces a more realistic depth than a single shadow.
    • A blur-radius of 0 with small offsets creates a solid "letterpress" or "emboss" effect. A large blur with no offset and a matching hue creates a glow effect around the text.
    • Contrast is critical — text-shadow alone should never substitute for sufficient color contrast between text and background. It can be used to improve legibility of light text on a complex image background, but always verify WCAG contrast ratios as well.
    /* Simple drop shadow on a heading */
h1 {
  text-shadow: 2px 2px 4px rgb(0 0 0 / 0.4);
}

/* Glow effect — zero offsets, wide blur, matching hue */
.neon {
  color: #7df9ff;
  text-shadow:
    0 0 4px  #7df9ff,
    0 0 16px #7df9ff,
    0 0 40px #00bfff;
}

/* Embossed / letterpress — tight shadow, no blur */
.embossed {
  color: #444;
  text-shadow: 0 1px 0 #fff;
}

/* White text on a complex image — shadow improves legibility */
.hero-caption {
  color: #fff;
  text-shadow: 0 1px 6px rgb(0 0 0 / 0.6);
}

    opacity

    • opacity sets the transparency of the entire element and all of its descendants as a group — values range from 0 (fully transparent) to 1 (fully opaque). Decimal and percentage values are both valid: opacity: 0.5 and opacity: 50% are equivalent.
    • A key distinction from rgba() color values: opacity affects the whole rendered element including its children, borders, and shadows as a single composited unit. Using background-color: rgba(0,0,0,0.5) makes only the background transparent while child text remains fully opaque; opacity: 0.5 makes everything — including child text — semi-transparent.
    • Any element with an opacity value less than 1 creates a new stacking context, the same way position + z-index does. This can affect the stacking order of overlapping elements.
    • opacity is a GPU-composited property — it is safe to animate and transition without causing layout recalculation, making it ideal for fade-in/fade-out effects.
    /* Disabled state — dims the whole button including its children */
.btn:disabled {
  opacity: 0.4;
  cursor: not-allowed;
}

/* Hover to full opacity — transition is GPU-composited */
.thumbnail {
  opacity: 0.8;
  transition: opacity 200ms ease;
}
.thumbnail:hover {
  opacity: 1;
}

/* NOTE: opacity affects ALL children.
   To make only the background transparent, use rgba() on the
   background-color instead: */
.overlay {
  background-color: rgb(0 0 0 / 0.6); /* background transparent */
  color: #fff;                          /* text stays fully opaque */
}

    CSS filter Functions

    The filter property applies graphical effects to an element and all of its content. Per MDN, filters are applied in the order they are listed — this matters because each function receives the output of the previous one, not the original element. Multiple filters are space-separated in a single filter declaration.

    • blur(radius) — applies a Gaussian blur. The argument must be a CSS length unit (px, rem, etc.) — percentages are not accepted. Larger values produce more blur. A value of 0 leaves the element unchanged. Commonly used for out-of-focus background effects.
    • brightness(amount) — multiplies the brightness of each pixel. 100% (or 1) is unchanged; 0% produces a fully black element; values above 100% are allowed and produce a brighter result. Accepts a number or percentage.
    • contrast(amount) — adjusts contrast. 100% is unchanged; 0% produces a flat grey; values above 100% increase contrast beyond the original.
    • grayscale(amount) — converts color to greyscale. 100% or 1 is fully greyscale; 0% is unchanged. Useful for desaturating images on hover or for a "disabled" visual treatment that avoids the flat look of opacity.
    • sepia(amount) — converts color to a warm brown sepia tone, like an old photograph. 100% or 1 is fully sepia; 0% is unchanged. Omitting the argument is equivalent to 100%.
    • hue-rotate(angle) — rotates every color on the element around the HSL color wheel by the given angle. 0deg is unchanged; 180deg inverts hues. Accepts deg, rad, and turn units. Useful for creating color-shifted variants of an image without editing the asset.
    • invert(amount) — inverts colors. 100% completely inverts; 0% is unchanged. A value of 100% on a greyscale image produces a photo-negative effect.
    • saturate(amount) — adjusts color saturation. 100% is unchanged; 0% removes all color (equivalent to grayscale(100%)); values above 100% produce an oversaturated result.
    • opacity(amount) — works like the opacity property. Per MDN, some browsers provide hardware acceleration specifically for filter: opacity(); in practice, for simple fades, the standalone opacity property is recommended as it is universally composited.
    • drop-shadow(offset-x offset-y blur color) — applies a shadow that follows the exact alpha channel of the element, not its bounding box. This is the key difference from box-shadow: on a PNG with a transparent background, drop-shadow() casts a shadow that traces the visible shape; box-shadow would shadow the full rectangle. Note that unlike box-shadow, drop-shadow() has no spread radius and no inset keyword.
    • Applying filter with any value other than none creates a new stacking context, the same way opacity does. Per MDN and CSS-Tricks, the filter has no effect on the element's box model geometry — it does not change layout, only painting.
    /* Single filter */
img.greyed {
  filter: grayscale(100%);
  transition: filter 300ms ease;
}
img.greyed:hover {
  filter: grayscale(0%);
}

/* Stack multiple filters — order matters.
   grayscale() receives the sepia() output, not the original. */
.vintage {
  filter: sepia(60%) contrast(110%) brightness(95%);
}

/* drop-shadow traces the PNG shape — box-shadow cannot do this */
.icon-png {
  filter: drop-shadow(4px 4px 6px rgb(0 0 0 / 0.4));
}

/* hue-rotate: shift colors across the wheel every frame */
@media (prefers-reduced-motion: no-preference) {
  .rainbow-icon {
    animation: hue 4s linear infinite;
  }
}
@keyframes hue {
  to { filter: hue-rotate(360deg); }
}

/* Combining blur + brightness for a frosted glass image overlay */
.hero-overlay {
  filter: blur(2px) brightness(0.7);
}

    backdrop-filter

    • backdrop-filter accepts the same filter functions as filter but applies them only to whatever is rendered behind the element — not to the element itself. Per MDN, the element must have a semi-transparent background (e.g. background: rgba(...) or background: transparent) for the backdrop effect to be visible. A fully opaque background would simply cover the blurred backdrop entirely.
    • The most common use case is the "glassmorphism" / frosted-glass card pattern: set a low-opacity white or dark background on the card along with backdrop-filter: blur(), and the content behind the card will appear softly blurred while the card's own text stays sharp.
    • Multiple functions are space-separated, exactly like filter. Combining blur() and saturate() is a particularly effective pairing — it mimics the frosted-glass look of macOS and iOS by blurring and slightly boosting the color saturation of the background.
    • Performance: backdrop blur is GPU-accelerated but can be expensive when applied to large elements on devices with limited graphics memory. Avoid applying it to full-screen elements or rapidly resizing containers. Per the W3Schools effects page, avoid heavy filters on large background videos or images.
    • Always provide a solid-color fallback background for browsers that don't support backdrop-filter. Use @supports (backdrop-filter: blur(1px)) { … } to conditionally enable the transparent backdrop version.
    /* Glassmorphism card — backdrop-filter blurs what's behind */
.glass-card {
  /* fallback for unsupported browsers */
  background: rgb(255 255 255 / 0.85);
  border-radius: 20px;
  border: 1px solid rgb(255 255 255 / 0.3);
}

@supports (backdrop-filter: blur(1px)) {
  .glass-card {
    /* Only applied when the browser supports it.
       The low-opacity background lets the blur show through. */
    background: rgb(255 255 255 / 0.12);
    backdrop-filter: blur(16px) saturate(120%);
  }
}

/* Dark frosted panel over a hero image */
.modal-overlay {
  background: rgb(15 23 42 / 0.5);
  backdrop-filter: blur(8px);
}

    Blend Modes

    CSS has two blend mode properties. Per MDN, background-blend-mode blends an element's background images and background color together; mix-blend-mode blends the element's entire rendered content with the content of its parent and the backdrop behind it.

    • background-blend-mode — set on an element that has multiple background layers (multiple background-image values and/or a background-color). Each value in a comma-separated list maps to the corresponding background layer. Common use: blend a color wash over a photo without needing to edit the image asset.
    • mix-blend-mode — set on an element to control how its painted output blends with whatever is behind it in the stacking context. It affects the whole element, including text and borders.
    • normal — no blending; the top element simply covers the elements beneath it. This is the default.
    • multiply — multiplies the color channels of the element and its backdrop. The result is always equal to or darker than either input. White areas become transparent; black areas stay black. Useful for placing a logo or illustration over a background without a white bounding box.
    • screen — the inverse of multiply. Multiplies the complements of both layers and complements the result. The output is always lighter. Black areas become transparent; white areas stay white. Good for light-coloured overlays and lens-flare effects.
    • overlay — combines multiply and screen: dark background areas get multiplied (darker), light areas get screened (lighter). Increases contrast while preserving highlights and shadows.
    • darken / lighten — selects the darker or lighter value per channel from the two blended layers. Simpler than multiply/screen and useful for subtle tonal adjustments.
    • difference — subtracts the darker color from the lighter one per channel. Produces a high-contrast inverted effect. White on the top layer inverts the bottom; black has no effect.
    • hue, saturation, color, luminosity — the component blend modes from the CSS Compositing spec. They mix only the named component from one layer with the remainder from the other, enabling effects like duotones (set hue from a solid color, luminosity from a photo) or desaturation overlays.
    • isolation: isolate — set on a container to create a new stacking context that prevents its children's mix-blend-mode from blending with elements outside the container. Per MDN, without isolation the blending extends to everything in the stacking context including ancestors, which can produce unintended results.
    /* background-blend-mode: multiply a red wash over a photo.
   The photo must be set as a background-image, not an . */
.photo-tint {
  background-color: #e63946;
  background-image: url("portrait.jpg");
  background-size: cover;
  background-blend-mode: multiply;
}

/* mix-blend-mode: knockout text through a background image.
   White text + multiply = the image shows through the letterforms. */
.knockout-text {
  background-image: url("texture.jpg");
  background-size: cover;
  background-clip: text;           /* clips the bg to the text shape */
  -webkit-background-clip: text;
  color: transparent;              /* makes the text colour invisible */
}

/* mix-blend-mode: screen — removes black from a logo.
   Any black areas become transparent, white areas stay white.
   Use on light-on-dark logos placed over a coloured background. */
.logo-light {
  mix-blend-mode: screen;
}

/* isolation: isolate — prevents blending from bleeding to the page */
.card-container {
  isolation: isolate; /* children blend only within this container */
}
.card-container .overlay {
  mix-blend-mode: overlay;
}

    Performance & Accessibility Checklist

    • Animate only opacity and transform for smooth, compositor-thread effects. Animating filter, box-shadow, or backdrop-filter directly can trigger paint recalculation every frame — if animation is needed, use the ::after opacity trick shown in the box-shadow examples above.
    • Apply backdrop-filter and filter: blur() to the smallest surface area possible. Large blurred surfaces are expensive on low-powered GPUs — avoid them on full-viewport backgrounds or inside frequently animated containers.
    • Always ensure text remains readable after applying effects. Filters that reduce contrast (brightness(), opacity()), blend modes that darken or wash out text, and heavy blurs behind poorly-contrasted text can all fail WCAG AA contrast requirements. Test with real content at real sizes.
    • Provide solid-color fallbacks for backdrop-filter using @supports. Firefox required a flag until version 103; ensure the fallback looks intentional rather than broken.
    • Blend modes and mix-blend-mode create new stacking contexts. If z-index layering behaves unexpectedly after applying a blend mode, check whether a new stacking context is the cause.

    Web Accessibility

    Accessibility means building pages that can be used by everyone — including people who navigate by keyboard, use screen readers, have low vision, or have cognitive disabilities. Most accessibility wins come from using HTML correctly and understanding a small set of rules. Per the W3Schools accessibility guide, the goal is to provide the user a good way to navigate and interact with your site using HTML that is as semantic as possible.

    Semantic HTML First

    • Semantic elements carry meaning about their content and role. Using the right element for the right purpose — <button> for buttons, <nav> for navigation, <main> for the primary content — gives assistive technologies the information they need to present that content correctly. Per web.dev, using semantic HTML means you do less work and have better-performing code because the browser handles keyboard interaction, focus, and ARIA semantics for you automatically.
    • Never use a <div> or <span> as a button or link. A <div> is not focusable by keyboard, does not fire click events on Enter or Space, and reports no role to screen readers. Using <button> gives you all three for free.
    • HTML landmark elements — <header>, <nav>, <main>, <aside>, <footer>, <section> — map directly to ARIA landmark roles. Per the W3Schools accessibility landmarks guide, screen reader users can jump directly between landmarks to navigate the page structure. Use only one <main> per page. When multiple landmarks of the same type exist (e.g. two <nav> elements), label each with aria-label so they can be distinguished.
    • Maintain a strict heading hierarchy: one <h1> per page, followed by <h2>s for major sections, then <h3>s for sub-sections. Screen reader users frequently navigate a page by jumping between headings — skipping levels (e.g. going from <h2> to <h4>) breaks that navigation model. Choose heading levels for document structure, not font size.
    <!-- Wrong: a div masquerading as a button. Not keyboard focusable,
     no role exposed, no Enter/Space activation. -->
<div class="btn" onclick="submit()">Submit</div>

<!-- Right: native button. Focusable, activatable by keyboard,
     announces "Submit, button" to screen readers. -->
<button type="button" onclick="submit()">Submit</button>

<!-- Multiple nav elements: label each so users can tell them apart -->
<nav aria-label="Primary">...</nav>
<nav aria-label="Breadcrumb">...</nav>
<nav aria-label="Footer">...</nav>

    ARIA — Roles, Names, and States

    WAI-ARIA (Accessible Rich Internet Applications) is a set of HTML attributes that supplement semantic meaning when native HTML elements are insufficient. Per the W3C WAI-ARIA overview, ARIA provides roles to describe widget types, properties to describe state, and a way to convey live updates. The first rule of ARIA use is: if you can use a native HTML element with the semantics and behaviour built in, do that instead.

    • role — overrides or supplies the element's semantic type as exposed to assistive technologies. Common landmark roles (role="navigation", role="main", role="banner", role="contentinfo") are automatically mapped from the corresponding HTML elements. Use explicit roles only when no suitable HTML element exists, or when you need to communicate a widget pattern not expressible in HTML alone (e.g. role="tablist", role="dialog", role="progressbar").
    • aria-label — provides an accessible name for an element when no visible text label is present or when the visible label is insufficient. Common uses: icon-only buttons (<button aria-label="Close dialog">), search inputs without a visible <label>, and landmark regions that need distinction. Per MDN, aria-label overrides any text content inside the element as the accessible name.
    • aria-labelledby — points to the id of another element whose text serves as the accessible name. Stronger than aria-label when a visible heading already describes the region: <section aria-labelledby="section-heading">. Multiple IDs can be space-separated to combine several elements into one accessible name.
    • aria-describedby — points to the id of an element that provides supplementary description (not the name). Used to associate hint text, error messages, or instructions with a form field. Screen readers announce the name first, then the description after a short pause.
    • aria-expanded — communicates whether a collapsible widget (accordion, dropdown, disclosure) is open or closed. Must be updated in JavaScript as the state changes. Per the W3Schools role/name/value guide, an accordion button needs aria-expanded="false" when collapsed and "true" when open so screen readers can announce the current state.
    • aria-hidden="true" — removes an element and all its descendants from the accessibility tree entirely. Use it on decorative icons, duplicated text, and elements that are visually hidden but not yet relevant. Never apply it to an element that is, or contains, focusable content — a keyboard user could tab into a hidden element and become stranded.
    • aria-live — marks a region whose content updates dynamically (toast notifications, chat messages, error banners) so screen readers announce changes without the user having to navigate to that region. aria-live="polite" waits for the user to finish their current action before announcing; aria-live="assertive" interrupts immediately and should be reserved for critical errors only.
    <!-- Icon button: aria-label provides the accessible name
     because there is no visible text -->
<button aria-label="Close dialog">
  <svg aria-hidden="true" focusable="false">...</svg>
</button>

<!-- Accordion: role + aria-expanded communicate widget state -->
<button
  aria-expanded="false"
  aria-controls="panel-1"
  id="header-1">
  When do I get charged?
</button>
<div id="panel-1" role="region" aria-labelledby="header-1" hidden>
  You are charged when the ride ends.
</div>

<!-- Live region: announces status messages to screen readers
     without requiring the user to navigate to it -->
<div aria-live="polite" aria-atomic="true" class="sr-only">
  <!-- Inject status text here via JavaScript -->
</div>

    Keyboard Navigation & tabindex

    Every interactive element on the page must be reachable and operable by keyboard alone — this is a hard WCAG Level A requirement (Success Criterion 2.1.1). Per the Grizzly Peak accessibility guide, the test is simple: unplug your mouse and use Tab, Shift+Tab, Enter, Space, and arrow keys to navigate. If you cannot complete every task, you have a problem.

    • Native interactive elements — <a href>, <button>, <input>, <select>, <textarea> — are keyboard-focusable and activatable by default with no extra work required. This is the single strongest argument for using semantic HTML over custom widgets.
    • tabindex="0" — inserts a non-interactive element into the natural tab order at its position in the DOM. Use this sparingly on custom interactive elements that are not natively focusable (e.g. a <div> with role="button"). If you find yourself doing this often, it is usually a sign that a native element should be used instead.
    • tabindex="-1" — removes an element from the tab order but keeps it programmatically focusable via JavaScript (element.focus()). This is the correct value for elements that should receive focus only under script control — modal dialog content, skip link targets, and drawer panels — not elements the user should tab through directly.
    • Never use positive tabindex values (e.g. tabindex="3"). Per the Grizzly Peak accessibility guide, positive values create a separate tab sequence that runs before the natural DOM order, producing a confusing navigation path that is nearly impossible to maintain as a page grows. If the DOM order is wrong, fix the DOM order.
    • When a modal or dialog opens, focus must move into it — typically to the first focusable element or the dialog's heading. When it closes, focus must return to the element that triggered it. This is called focus management and is required for modals to be usable by keyboard and screen reader users.
    • Keyboard traps — situations where a keyboard user cannot move focus out of a component — are a WCAG 2.1.2 failure. The one exception is intentional modal trapping: focus must be contained within an open modal so the user cannot interact with background content while it is open, but Escape must always close the modal and return focus.
    /* Visible focus styles: never remove the outline without replacing it.
   :focus-visible shows the ring only for keyboard navigation,
   not on mouse clicks, satisfying both visual design and accessibility. */
:focus-visible {
  outline: 3px solid #2563eb;
  outline-offset: 3px;
  border-radius: 4px;
}

/* Remove the default outline only after providing a custom one */
:focus:not(:focus-visible) {
  outline: none;
}

/* Utility class for screen-reader-only text.
   Positions content off-screen rather than using display:none
   so it remains in the accessibility tree and receives focus. */
.sr-only {
  position: absolute;
  width: 1px;
  height: 1px;
  padding: 0;
  margin: -1px;
  overflow: hidden;
  clip: rect(0, 0, 0, 0);
  white-space: nowrap;
  border: 0;
}

    Skip Links

    • A skip link is an anchor placed as the very first focusable element on the page that, when activated, moves focus past the navigation to the main content. It prevents keyboard and screen reader users from having to tab through every nav item on every page load. Per the W3Schools accessibility guide, it is one of the most impactful single fixes you can make.
    • Skip links are typically visually hidden by default and revealed only when focused — this is acceptable because sighted keyboard users will see them when they tab to them, while mouse users never need them.
    • The skip link's href must point to an element with a matching id. That target element should have tabindex="-1" applied so that when the skip link sends focus there, it actually lands on the element rather than being skipped. Without tabindex="-1", some browsers do not move keyboard focus even though the viewport scrolls.
    <!-- First focusable element in the document -->
<a href="#main-content" class="skip-link">Skip to main content</a>

<nav aria-label="Primary">...</nav>

<!-- tabindex="-1" ensures focus lands here when the skip link
     is activated, even in browsers that don't natively focus
     non-interactive elements -->
<main id="main-content" tabindex="-1">...</main>

/* CSS: hide visually, reveal on focus */
.skip-link {
  position: absolute;
  top: -100%;
  left: 0;
  padding: 12px 20px;
  background: #2563eb;
  color: #fff;
  font-weight: 700;
  z-index: 9999;
  text-decoration: none;
  border-radius: 0 0 8px 0;
  transition: top 0.1s ease;
}
.skip-link:focus {
  top: 0;
}

    Color Contrast & Color Meaning

    • WCAG 2.1 Success Criterion 1.4.3 (Level AA) requires a minimum contrast ratio of 4.5:1 between text and its background for body text, and 3:1 for large text (defined as 18pt normal weight or 14pt bold and above). Per the W3Schools accessibility page, test every text and background color pair with a contrast checker before shipping.
    • WCAG 2.1 Success Criterion 1.4.11 (Level AA) extends the 3:1 requirement to non-text contrast — interactive UI components (button borders, input outlines, focus indicators) and meaningful graphical elements must also meet a 3:1 ratio against adjacent colors.
    • Color must never be the only way information is conveyed — this is WCAG 1.4.1. Per the W3Schools accessibility page, if you use red to indicate a required form field or green to indicate success, you must also add an icon, a text label, or a pattern so the information is available to users who are color-blind or who cannot distinguish those hues. Approximately 8% of men and 0.5% of women have some form of color vision deficiency.
    • Use browser DevTools' accessibility panel or dedicated tools like the WebAIM Contrast Checker to audit contrast ratios. The forced-colors: active media query can be used in CSS to provide fallback styles for Windows High Contrast Mode, where the OS overrides all colors.
    /* Accessible focus ring that meets 3:1 non-text contrast */
:focus-visible {
  outline: 3px solid #1d4ed8; /* verified 4.5:1+ against white */
  outline-offset: 3px;
}

/* Error state: color + icon + text label — not color alone */
.field-error input {
  border-color: #dc2626;
}
.field-error .hint::before {
  content: "⚠ "; /* icon supplements the color cue */
}
.field-error .hint {
  color: #dc2626;
  font-weight: 600;
}

/* High Contrast Mode: use system color keywords instead of
   fighting the OS override */
@media (forced-colors: active) {
  .btn {
    border: 2px solid ButtonText;
  }
}

    Images, Video & Audio

    • Every <img> must have an alt attribute — omitting it entirely is invalid HTML and causes screen readers to announce the filename instead. Per the W3Schools accessibility page, alt text should explain the purpose of the image in context, not just describe what it looks like. Ask: if this image were removed, what information would be lost?
    • Decorative images that add no information — dividers, background textures, purely aesthetic illustrations — should use an empty alt="". This tells screen readers to skip the element entirely. Do not use a space (alt=" ") — some screen readers announce the space as content.
    • Complex images such as charts, diagrams, and infographics need a more complete description than a short alt string can provide. Per MDN, use a <figure> with a <figcaption> for a visible long description, or use aria-describedby to link the image to a separate descriptive paragraph.
    • For <video> elements, provide closed captions via a <track kind="captions"> element for users who are deaf or hard of hearing, and a text transcript for users who cannot receive any audio or video. For audio-only content, a transcript is the primary accessibility requirement.
    • Never use images of text — render text in HTML and CSS instead. Images of text cannot be resized cleanly, cannot be selected, cannot be translated, and fail contrast testing when the image is compressed.
    <!-- Informative image: alt describes the purpose -->
<img src="sales-chart.png"
     alt="Bar chart showing Q3 revenue up 18% year over year">

<!-- Decorative image: empty alt tells screen readers to skip it -->
<img src="divider-wave.svg" alt="">

<!-- Complex image: short alt + long description via figcaption -->
<figure>
  <img src="network-diagram.png"
       alt="Network architecture — full description below">
  <figcaption>
    Three tiers: a CDN edge layer, an application load balancer,
    and two redundant app servers connected to a primary and
    replica database pair.
  </figcaption>
</figure>

<!-- Video with captions track -->
<video controls>
  <source src="demo.mp4" type="video/mp4">
  <track kind="captions" src="captions-en.vtt"
         srclang="en" label="English" default>
</video>

    Accessible Forms

    • Every <input>, <select>, and <textarea> must have a programmatically associated <label>. Link them with matching for and id attributes. Per the W3Schools accessibility guide, never use placeholder alone as a label — placeholders disappear as soon as the user starts typing and fail the persistence requirement for accessible labels.
    • Group related controls — radio buttons, checkboxes — inside a <fieldset> with a <legend> as its first child. The legend provides the group question that gives each individual option its context. Without it, a screen reader user hears "Yes, radio button" with no way to know what the question was.
    • Use aria-describedby to associate hint text or format instructions with an input — screen readers will announce the hint after the label. Use aria-invalid="true" together with an error message linked via aria-describedby when a field fails validation. Do not rely on color or position alone to communicate errors.
    • Use the correct type attribute on inputs. type="email", type="tel", and type="number" trigger the appropriate mobile keyboard and help password managers and autofill identify the field's purpose. The autocomplete attribute enables browser and OS autofill — use values like "email", "name", "current-password" to map fields to their purpose (WCAG 1.3.5).
    • Never disable browser zoom. The viewport meta tag maximum-scale=1 or user-scalable=no prevents users with low vision from enlarging the page and is a WCAG 1.4.4 failure. Per the Grizzly Peak accessibility guide, users with low vision need to zoom — do not take that away from them.
    <!-- Correct label association + hint text via aria-describedby -->
<label for="email">Email address</label>
<input type="email"
       id="email"
       name="email"
       autocomplete="email"
       aria-describedby="email-hint">
<span id="email-hint">We'll only use this to send your receipt.</span>

<!-- Validation error: aria-invalid + linked error message -->
<label for="username">Username</label>
<input type="text"
       id="username"
       aria-invalid="true"
       aria-describedby="username-error">
<span id="username-error" role="alert">
  Username must be at least 4 characters.
</span>

<!-- Grouped radio buttons with fieldset + legend -->
<fieldset>
  <legend>Preferred contact method</legend>
  <label><input type="radio" name="contact" value="email"> Email</label>
  <label><input type="radio" name="contact" value="phone"> Phone</label>
</fieldset>

<!-- Never do this — it locks out low-vision users -->
<!-- meta name="viewport" content="width=device-width, maximum-scale=1" -->

    Design Systems

    A design system unifies design and development with shared foundations, reusable components, and documented standards. HTML and CSS provide the blueprint for consistent interfaces across an entire product.

    Design Tokens

    • Design tokens are named values — such as --color-primary or --spacing-lg — that encode every decision about color, typography, spacing, border-radius, and elevation into a single source of truth. Storing them as CSS custom properties means a one-line change updates every component that references the token, instead of a search-and-replace across dozens of files.
    • Define all tokens at :root so they cascade to every element. Group them by category in a dedicated tokens.css file that is loaded first in dependency order. For cross-platform use (web + mobile + design tools), maintain the canonical token values as JSON and generate the CSS and platform-specific outputs from that single source via a token pipeline.
    • Document responsive breakpoints, grid column counts, and gutter sizes as tokens too — this prevents hardcoded magic numbers from leaking into component stylesheets. Motion tokens (--duration-fast: 150ms, --easing-standard: ease-out) ensure every transition and animation across the system feels consistent.
    /* tokens.css — loaded first, referenced everywhere */
:root {
  /* Color */
  --color-primary:   #1d4ed8;
  --color-surface:   #ffffff;
  --color-text:      #111827;
  --color-muted:     #6b7280;

  /* Spacing scale */
  --space-1:  0.25rem;
  --space-2:  0.5rem;
  --space-4:  1rem;
  --space-8:  2rem;

  /* Typography */
  --font-sans:   system-ui, sans-serif;
  --text-base:   1rem;
  --text-lg:     1.125rem;
  --leading-normal: 1.6;

  /* Radius */
  --radius-sm: 4px;
  --radius-md: 8px;
  --radius-full: 9999px;

  /* Motion */
  --duration-fast:  150ms;
  --easing-standard: ease-out;
}

/* A component consuming tokens — no hardcoded values */
.btn-primary {
  background: var(--color-primary);
  padding: var(--space-2) var(--space-4);
  border-radius: var(--radius-md);
  font-family: var(--font-sans);
  transition: background var(--duration-fast) var(--easing-standard);
}

    Component Library

    • Each component should be a self-contained HTML + CSS unit: semantic markup as the structure, scoped CSS for styles, and a well-documented API of modifier classes for variants. A .btn base class handles the shared styles; .btn--secondary, .btn--sm, and .btn--danger layer on variations without duplicating code.
    • Include accessibility notes alongside every component: which ARIA roles and attributes it requires, what keyboard interactions it must support (Enter/Space for buttons, arrow keys for listboxes), and whether it needs a visible label or an aria-label fallback. A component with no accessibility spec is incomplete.
    • Provide states explicitly: default, hover, focus, active, disabled, loading, and error where applicable. Disabled elements should use the HTML disabled attribute (not just a CSS class) so they are excluded from focus order and correctly announced by screen readers. Error states must pair color with a text or icon cue — never color alone.
    <!-- Base + modifier class pattern -->
<button class="btn btn--primary">Save</button>
<button class="btn btn--secondary">Cancel</button>
<button class="btn btn--primary btn--sm">Compact</button>
<button class="btn btn--primary" disabled>Submitting…</button>

/* Component CSS — all values pull from tokens */
.btn {
  display: inline-flex;
  align-items: center;
  gap: var(--space-2);
  padding: var(--space-2) var(--space-4);
  border: 2px solid transparent;
  border-radius: var(--radius-md);
  font-family: var(--font-sans);
  font-size: var(--text-base);
  cursor: pointer;
  transition: background var(--duration-fast) var(--easing-standard),
              outline-offset 80ms ease;
}

  outline: 3px solid var(--color-primary);
  outline-offset: 3px;
}

  opacity: 0.45;
  cursor: not-allowed;
}
.btn--primary  { background: var(--color-primary); color: #fff; }
.btn--secondary { background: transparent; border-color: var(--color-primary); color: var(--color-primary); }
.btn--sm { padding: var(--space-1) var(--space-2); font-size: 0.875rem; }

    Documentation & Governance

    • Every component in the library needs written documentation: what it is for, when to use it versus an alternative, explicit do/don't examples, the full list of modifier classes and their effects, and copy-paste code snippets. Without documentation, the library is only usable by the people who built it.
    • Host the living documentation with a tool like Storybook (renders components in isolation with interactive controls), Zeroheight (bridges Figma designs to coded components), or a custom static site. The hosted docs should always reflect the latest published version of the library.
    • Establish a contribution and review process: how to propose a new component, who reviews it, what checklist it must pass (accessibility audit, cross-browser check, token usage, documentation) before it is merged. Version the library with semantic versioning — breaking changes should never land silently. Automate visual regression tests with tools like Chromatic or Percy to catch unintended CSS drift between releases.

    Performance

    Page performance directly affects user experience and search rankings. The browser must download, parse, and render HTML, CSS, fonts, and images before the user sees a usable page — every unnecessary byte and blocking resource adds latency.

    Critical Rendering Path

    • The browser cannot render anything until it has built both the DOM (from HTML) and the CSSOM (from CSS). CSS is render-blocking by default — a stylesheet linked in <head> halts rendering until it is fully downloaded and parsed. Keep your CSS lean, load it early, and split large stylesheets into smaller files that can be conditionally loaded with media attributes (e.g., media="print" for print styles).
    • JavaScript without defer or async is also render-blocking and parser-blocking: the HTML parser stops, the script downloads, executes, then parsing resumes. Use defer for scripts that need the full DOM (they execute after parsing, in order). Use async only for truly independent scripts (analytics, ad tags) that do not depend on DOM order. Place all scripts at the end of <body> or in <head> with defer — never in the middle of the body without one of these attributes.
    • Inline the minimum CSS required to render above-the-fold content inside a <style> block in <head>. This eliminates the render-blocking network request for that critical CSS. Load the full stylesheet asynchronously with a <link rel="preload"> trick or a JavaScript-based loader. Keep inlined CSS to an absolute minimum — usually 1–3 KB uncompressed.
    <!-- Preload the main font so it's ready before text renders -->
<link rel="preload" href="/fonts/inter.woff2" as="font"
      type="font/woff2" crossorigin>

    Images & Asset Optimization

    • Choose the right format for every image. Use WebP or AVIF for photos and complex graphics — they are 25–50% smaller than JPEG at equivalent quality and are supported in all modern browsers. Use SVG for icons, logos, and illustrations — SVGs are resolution-independent and can be styled with CSS. Use <picture> with <source type="image/avif"> and <source type="image/webp"> falling back to a <img> JPEG/PNG for maximum compatibility.
    • Add loading="lazy" to every <img> that is not in the initial viewport. The browser will defer downloading the image until the user scrolls close to it. Do not lazy-load the Largest Contentful Paint (LCP) image — typically the hero image — as this delays the most important visual element and hurts Core Web Vitals scores.
    • Always specify width and height attributes on <img> matching the image's natural dimensions. The browser uses these to reserve the correct amount of space before the image loads, eliminating Cumulative Layout Shift (CLS) — the jarring jump that occurs when images pop in and push content down.
    • Minify and compress all CSS, HTML, and JavaScript files before deploying. Enable Gzip or Brotli compression on your server — Brotli typically achieves 15–25% better ratios than Gzip. Use a CDN to serve assets from edge nodes geographically close to users, reducing round-trip time.
    <!-- Optimal image element: modern format, lazy, explicit dimensions -->
<picture>
  <source srcset="hero.avif" type="image/avif">
  <source srcset="hero.webp" type="image/webp">
  <img src="hero.jpg"
       alt="A developer at a desk writing code"
       width="1200" height="630"
       loading="eager">  <!-- hero: eager, not lazy -->
</picture>

<!-- Below-fold images: lazy-load to defer network requests -->
<img src="feature.webp"
     alt="Screenshot of the dashboard"
     width="800" height="450"
     loading="lazy">

    CSS Performance Patterns

    • Prefer transform and opacity for all animations and transitions. These two properties are handled entirely by the GPU compositor and never trigger layout or paint. Animating width, height, top, left, margin, or padding forces the browser to recalculate layout on every frame, which is expensive and causes visible jank on lower-powered devices.
    • Avoid overly broad selectors that force the browser to evaluate many elements. Descendant combinators like div p span are evaluated right-to-left — the browser first finds every <span>, then checks if each one is inside a <p>, then inside a <div>. A single class like .card-label is faster and more maintainable.
    • Use content-visibility: auto on tall, off-screen sections (e.g., long article bodies, comment threads). The browser skips layout and paint for those sections until they approach the viewport, dramatically reducing initial render time on content-heavy pages. Pair it with contain-intrinsic-size to provide an estimated height so the scrollbar doesn't jump as sections are rendered.
    /* GPU-composited: no layout recalculation on every frame */
.slide-in {
  transition: transform 300ms ease-out, opacity 300ms ease-out;
}
.slide-in.is-hidden {
  transform: translateY(20px);
  opacity: 0;
}

/* Defer rendering of long off-screen content */
.article-body {
  content-visibility: auto;
  contain-intrinsic-size: 0 600px; /* estimated height */
}

/* Avoid: triggers layout recalculation each frame */
/* .panel { transition: height 0.3s ease; } */
/* Use max-height trick or CSS Grid row transitions instead */

    Debugging & Testing

    Finding and fixing HTML/CSS problems efficiently requires the right tooling, a systematic approach, and a set of techniques for isolating the source of broken layouts, unexpected specificity, or visual regressions.

    Browser DevTools

    • The Elements panel (Chrome/Edge) or Inspector (Firefox) is the primary tool for HTML and CSS debugging. Click any element to see every CSS rule applied to it in source order, with overridden rules crossed out. The Computed tab shows the final resolved value for every property, including inherited values — indispensable when a property value is not what you expect.
    • The Styles panel lets you add, edit, and toggle declarations live without touching source files. Toggle individual declarations with the checkbox beside each rule to isolate which CSS is causing a problem. Add declarations directly in element.style to test a fix before writing it into a stylesheet. Use the :hov button to force :hover, :focus, and other pseudo-class states without interacting with the page.
    • In Chrome DevTools, open Layout in the Elements sidebar to visualize Flexbox and Grid overlays. Clicking the grid/flex icon beside a container in the Styles panel draws the track lines and gap areas directly over the page, making alignment and overlap issues immediately obvious.
    • The Accessibility tree panel shows exactly what screen readers see — the computed role, name, and state of every element. Use it to verify that aria-label, aria-labelledby, and role assignments are working as intended before testing with an actual screen reader.
    /* DevTools trick: temporary red outline on every element
   to visualize the box model and spot overflow */
* { outline: 1px solid red !important; }

/* Or: highlight only overflowing elements */
* { overflow: visible !important; outline: 2px dashed magenta; }

/* Useful while debugging: make all elements' boxes visible */
.debug * {
  background: rgb(255 0 0 / 0.05) !important;
  outline: 1px solid rgb(255 0 0 / 0.3) !important;
}

    HTML & CSS Validation

    • Run your HTML through the W3C Markup Validator regularly. Many layout bugs are caused by invalid nesting — a <div> inside a <p>, a <td> without a parent <tr>, or unclosed tags — that browsers silently auto-correct in unpredictable ways. The validator surfaces these before they become hard-to-diagnose rendering inconsistencies across different browsers.
    • For CSS, use the W3C CSS Validation Service or enable CSS Lint / Stylelint in your editor. Stylelint can enforce no-duplicate-selectors, no-descending-specificity, property ordering conventions, and token usage rules — catching problems at write-time rather than in the browser.
    • Run Lighthouse (built into Chrome DevTools under the Audits/Lighthouse tab) to get scored reports on performance, accessibility, best practices, and SEO. Each failed audit includes a direct explanation and a link to the relevant guidance. Fix Lighthouse issues before deploying — they surface missing alt text, insufficient contrast ratios, missing viewport tags, render-blocking resources, and more.

    Cross-Browser & Responsive Testing

    • Use the DevTools device toolbar (Ctrl/Cmd+Shift+M in Chrome) for rapid responsive testing — it emulates viewport dimensions and touch events. However, emulation is not the same as a real device: test on physical hardware for real performance, font rendering, and touch behavior whenever possible.
    • Check your pages in at least Chrome, Firefox, and Safari. Safari (WebKit) regularly lags on implementing newer CSS features and has historically been the source of flexbox, grid, and scroll-related bugs. Use caniuse.com to check browser support before using any CSS property not in your pre-existing toolkit.
    • For systematic cross-browser and cross-device testing, services like BrowserStack or LambdaTest provide real browser instances in the cloud. Use visual regression tools (Chromatic, Percy, or Playwright's screenshot comparison) in CI to automatically flag layout changes between code deployments.

    Tooling & Automation

    Modern front-end tooling automates repetitive tasks — compiling, bundling, linting, formatting, and deploying — so you can focus on writing good HTML and CSS rather than managing files by hand.

    CSS Preprocessors & PostCSS

    • Sass/SCSS extends CSS with variables, nesting, mixins, functions, and partials (files prefixed with _ that are imported into a main file). Sass compiles to plain CSS before the browser ever sees it. Most of what Sass offered for variables and nesting is now possible in plain CSS with custom properties and native nesting — but Sass remains valuable for its @each loops, @mixin/@include patterns, and large project organization tools like @use and @forward.
    • PostCSS is a CSS transformation tool powered by plugins. Autoprefixer automatically adds vendor prefixes (-webkit-, -moz-) based on a target browser list from browserslist, so you never have to write them by hand. postcss-preset-env lets you write future CSS syntax today and compiles it down to what target browsers currently support — similar to Babel for JavaScript.
    • CSS Modules (used with bundlers like Webpack or Vite) automatically scope all class names to the component file that defines them, eliminating global namespace collisions without any naming convention discipline. The class .button in Card.module.css becomes something like .Card_button__abc12 in the output, guaranteeing it will never clash with a .button in another file.
    /* SCSS: variables, nesting, and a mixin in one block */
$primary: #1d4ed8;
$radius: 8px;

@mixin focus-ring($color: $primary) {
  &:focus-visible {
    outline: 3px solid $color;
    outline-offset: 3px;
  }
}

.btn {
  background: $primary;
  border-radius: $radius;
  @include focus-ring();

  &:hover { background: darken($primary, 10%); }
  &--sm   { padding: 0.25rem 0.5rem; font-size: 0.875rem; }
}

    Bundlers & Build Tools

    • Vite is the current dominant choice for new projects. It serves files via native ES modules during development (no bundling step — near-instant HMR), then uses Rollup for an optimized production build. Its CSS handling is built-in: it processes SCSS, PostCSS, and CSS Modules out of the box with minimal configuration.
    • Webpack remains common in large, existing codebases due to its mature ecosystem. CSS is processed through css-loader (resolves @import and url()) and style-loader (injects styles into the DOM) or MiniCssExtractPlugin (extracts CSS to a separate file for production). Configuration is verbose compared to Vite but extremely flexible.
    • For pure static sites or simple projects, a lightweight approach using PostCSS CLI with Autoprefixer, a CSS minifier like cssnano, and an HTML minifier is perfectly sufficient — no bundler required. Add a package.json scripts block to chain these commands and run them with npm run build.
    // vite.config.js — minimal setup with SCSS and PostCSS
import { defineConfig } from 'vite';

export default defineConfig({
  css: {
    preprocessorOptions: {
      scss: { additionalData: '@use "src/tokens" as *;' }
    },
    postcss: './postcss.config.js'
  }
});

// postcss.config.js
export default {
  plugins: [
    require('autoprefixer'),
    require('cssnano')({ preset: 'default' })
  ]
};

    Linting, Formatting & CI

    • Stylelint lints CSS (and SCSS) for errors and convention violations. Configure it with .stylelintrc.json — useful rules include no-duplicate-selectors, no-descending-specificity, color-no-invalid-hex, declaration-block-no-shorthand-property-overrides, and a custom custom-properties-pattern to enforce your token naming convention. Run it in CI so non-conforming CSS cannot be merged.
    • Prettier formats HTML and CSS automatically and consistently on every save. Pair it with an editor plugin (VS Code's Prettier extension) and a pre-commit hook via lint-staged + Husky so code is always formatted before it enters version control. This eliminates style debates in code review entirely — formatting is never a human decision.
    • A minimal CI pipeline for a CSS-heavy project should run: install → lint (Stylelint + HTML validator) → build → visual regression test → deploy to a preview URL for review. Store the pipeline in .github/workflows/ci.yml (GitHub Actions) or an equivalent. Every pull request should produce a preview deployment so reviewers can check the visual output in a real browser, not just read a diff.
    # .github/workflows/ci.yml — minimal HTML/CSS CI pipeline
name: CI
on: [push, pull_request]

jobs:
  build-and-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 20 }
      - run: npm ci
      - run: npm run lint        # stylelint + eslint
      - run: npm run build       # vite build
      - run: npm run test:visual # playwright screenshot diff

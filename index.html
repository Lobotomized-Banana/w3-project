Head Metadata

  • <meta charset="UTF-8"> — declares the character encoding. UTF-8 covers virtually all characters and should always be the first element inside <head>.
  • <meta name="viewport" content="width=device-width, initial-scale=1"> — required for responsive layouts on mobile. Without it, mobile browsers render at a desktop width and scale down.
  • <link rel="stylesheet" href="path/to/file.css"> — attaches an external CSS file. Place stylesheet links in <head> so the page doesn't flash unstyled content (FOUC).
  • <script src="file.js" defer></script> — the defer attribute delays script execution until HTML parsing is complete, preventing render blocking. Use async for scripts with no dependencies. Scripts without either attribute block parsing when encountered.
  • <title> — required element. Sets the browser tab label and is used by search engines as the page headline. Should be unique per page, ideally 50–60 characters.
  • <meta name="description" content="..."> — provides a summary shown as the snippet in search results. Recommended length is 150–160 characters.

Layout containers

  • <header> — introductory content for its nearest sectioning ancestor (page or section). Typically holds the site logo, branding, and primary navigation. There can be more than one per page as long as they're scoped to different sections.
  • <nav> — wraps a major block of navigation links. Use aria-label to distinguish multiple <nav> elements on the same page (e.g., aria-label="Primary", aria-label="Footer").
  • <main> — contains the dominant, unique content of the page. There must be only one <main> per page, and it should not be nested inside <article>, <aside>, <header>, <footer>, or <nav>.
  • <section> — a thematically grouped block that should have its own heading (h2–h6). Use <article> instead when the content could stand independently (e.g., a blog post). Use <aside> for tangentially related content like sidebars.
  • <footer> — closing information for its nearest sectioning ancestor. Typically holds copyright, secondary links, and contact info. Like <header>, multiple footers are valid when scoped to different sections.

Organizing CSS

  • Link external stylesheets in <head> with <link rel="stylesheet" href="...">. The browser downloads and applies them before rendering the page, preventing a flash of unstyled content.
  • For critical CSS (above-the-fold styles needed immediately), inline it inside a <style> tag in <head> to avoid a render-blocking network request. Keep this to a minimum.
  • Use <style> blocks for page-specific overrides. Avoid inline style="" attributes except for values that are genuinely dynamic or must be guaranteed to load with the element.
  • Load multiple stylesheets in dependency order — resets and base styles first, then component styles, then page-specific overrides.

Accessibility hooks

  • <html lang="en"> — the lang attribute is required for screen readers to select the correct voice/pronunciation engine, and for browser translation tools. Use a valid BCP 47 language tag (e.g., en, fr, pt-BR).
  • Every <section>, <nav>, <aside>, and <region> landmark should have an accessible name — either via a visible heading (h2–h6) or an aria-label/aria-labelledby attribute. Without one, screen readers announce the element type with no context.
  • Maintain heading hierarchy in document order: one <h1> per page, followed by <h2>s, then <h3>s, etc. Skipping levels (e.g., going from h2 to h4) breaks navigation for screen reader users who jump between headings.
  • All interactive elements must be keyboard-reachable and show a visible :focus indicator. Never remove the default outline without replacing it with an equally visible alternative.
  • Color alone should never be the only way information is conveyed — always pair color cues with text, icons, or patterns.

Text Content

  • Use exactly one <h1> per page — it names the page's primary topic. Follow with <h2> for major sections and <h3>–<h6> for subsections. Heading levels convey document structure to both screen readers and search engines; don't choose heading levels based on font size alone — use CSS for that.
  • <p> wraps paragraphs of text. Block-level element; cannot be nested inside itself or other block elements like <div> that are already inside a <p>.
  • <br> inserts a line break within inline content (e.g., inside a poem or address). Use sparingly — do not use it to create spacing between paragraphs; use margin on <p> elements instead.
  • <hr> represents a thematic break between paragraph-level content (e.g., a scene change in a story). It is a void element with no closing tag.
  • <strong> — marks text with strong importance. Renders as bold by default and carries semantic meaning (important to screen readers). Use font-weight: bold in CSS if you only want the visual effect.
  • <em> — marks text with stress emphasis. Renders as italic by default and affects how screen readers read the sentence aloud. Use font-style: italic in CSS for purely visual italics.
  • <mark> — highlights text as relevant in the current context (e.g., search result matches). Not the same as importance; use <strong> for importance.
  • <code> — inline code snippet, rendered in a monospace font. For multi-line code blocks, wrap <code> inside <pre> to preserve whitespace and line breaks.
  • <abbr title="..."> exposes the full expansion of an abbreviation on hover and to assistive technology. <cite> marks the title of a referenced work. <time datetime="YYYY-MM-DD"> encodes dates and times in a machine-readable format.
  • For readability, set line-height: 1.5–1.7 on body text, use generous paragraph margin-bottom, and keep line lengths between 60–80 characters using max-width on text containers.

Basic Lists

  • <ul> — unordered list. Use when the sequence of items does not matter. Renders with bullet points by default.
  1. <ol> — ordered list. Use when sequence matters (steps, rankings). Renders with numbers by default. Accepts start (e.g., start="3") to begin numbering from a value other than 1, and reversed to count down.
  • All list items must be wrapped in <li>. The only valid direct children of <ul> and <ol> are <li> elements — placing other elements directly inside the list is invalid HTML.
  • Lists can be nested by placing a new <ul> or <ol> inside an <li>. Style with list-style, padding, gap, and flexbox or grid on the <ul>/<ol> container.

Description List

  • <dl> — description list. Used for groups of term–definition pairs such as glossaries, metadata, or FAQs. The only valid direct children are <dt> and <dd> (and optionally <div> to group pairs for styling).
  • <dt> — the term being defined. Multiple <dt> elements in a row are valid and indicate that the following <dd> applies to all of them.
  • <dd> — the description or value for the preceding <dt>. Multiple <dd> elements can follow a single <dt> for multiple definitions.

Links & nav

  • <a href="url"> — creates a hyperlink. href is required for the element to be interactive. Use relative paths for internal links (href="about.html") and absolute URLs for external ones.
  • For external links, always add target="_blank" rel="noopener noreferrer". target="_blank" opens in a new tab; noopener prevents the new page from accessing window.opener (a security risk); noreferrer also suppresses the HTTP referrer header.
  • title="..." adds an optional tooltip on hover. Don't rely on it for critical information — it's inaccessible to touch and keyboard users.
  • download — prompts the browser to download the linked file rather than navigate to it. Can optionally take a filename value: download="filename.pdf". Only works for same-origin URLs or data URIs.
  • Link text should describe the destination — avoid generic labels like "click here" or "read more," which are meaningless out of context for screen reader users.

Nav menus

  • Wrap navigation links in a <ul> inside <nav> — this provides list semantics so screen readers announce how many items are in the menu. Each link goes in an <li><a href="...">.
  • Add aria-label to <nav> when there are multiple nav regions on the page so they can be distinguished (e.g., aria-label="Primary", aria-label="Breadcrumb").
  • Mark the current page's link with aria-current="page" so screen readers identify it. An .active CSS class handles the visual style, but aria-current handles the semantic meaning.

States and Transitions

  • Add an .active class (or aria-current="page") to the nav link for the current page so users always know where they are.
  • :hover styles interactive elements on mouse-over. :focus applies when an element is focused via keyboard or programmatically. :focus-visible is preferred — it only shows the focus ring when the keyboard (not the mouse) is used, preventing the ring from appearing on mouse clicks while keeping it for keyboard navigation. Never set outline: none on :focus without a :focus-visible replacement.
  • Use transition: property duration easing; on the element's base state, not on the :hover rule, so the transition plays in both directions. Example: transition: background-color 0.2s ease;

Responsive Menus

  • On mobile, convert horizontal navigation into a vertical stack using a media query and flex-direction: column, or hide it behind a toggle button.
  • For a hamburger/toggle menu, use a <button> (not a <div> — buttons are keyboard-focusable and fire click events on Enter/Space by default). Toggle a class on the nav with JavaScript to show/hide it. Add aria-expanded="true/false" on the button to communicate the menu state to screen readers.
  • When the menu is hidden, use display: none or visibility: hidden rather than just moving it off-screen, so keyboard users cannot tab into hidden links.

Images and Media

  • <img src="url" alt="description"> — src and alt are both required. alt must describe the image's content or function. For decorative images that add no information, use an empty alt="" so screen readers skip it. Omitting alt entirely is invalid.
  • Set width and height attributes on <img> to match the image's natural dimensions — this lets the browser reserve space before the image loads, preventing layout shift. In CSS, use max-width: 100% and height: auto to keep it responsive within that reserved space.
  • <picture> with <source> elements allows serving different image files based on viewport size or format support. Each <source> uses a media attribute for breakpoints and srcset for the file path. The <img> inside <picture> is the required fallback and is where alt lives.
  • srcset on a plain <img> lets you provide multiple resolution variants (e.g., srcset="img-400.jpg 400w, img-800.jpg 800w") and the browser picks the best one based on device pixel ratio and viewport width.
  • <video> and <audio> support native browser playback. Common attributes: controls (shows playback UI — required for user accessibility), autoplay (use with muted, browsers block autoplay with sound), loop, preload. The poster attribute on <video> sets a thumbnail image shown before playback. Nest <source type="video/mp4"> elements inside for multiple formats, and include fallback text for browsers that don't support the element.
  • For YouTube/Vimeo embeds, wrap the <iframe> in a container with position: relative; padding-bottom: 56.25%; (56.25% = 9/16 for a 16:9 ratio), then set the iframe to position: absolute; top: 0; left: 0; width: 100%; height: 100%; to maintain aspect ratio at any width.
  • Add loading="lazy" to <img> elements below the fold to defer their network request until they're near the viewport. Do not use it on images above the fold — it delays images the user sees immediately.
  • Wrap images with captions in <figure> with a <figcaption> child. This semantically associates the caption with the image. <figure> can also wrap code blocks, charts, or other self-contained content.

Forms and Validation

  • <form action="url" method="post"> — action is where the form data is submitted; method is either GET (data appended to URL, suitable for searches) or POST (data in request body, required for sensitive data or large payloads).
  • Add autocomplete="on" (or field-specific values like autocomplete="email") to help browsers and password managers fill fields correctly. Add novalidate to the form element if you are handling validation entirely with JavaScript.
  • Every <input>, <select>, and <textarea> must have an associated <label>. Link them with a matching id on the input and for on the label: <label for="email"> / <input id="email">. Clicking the label then focuses the input. Do not rely solely on placeholder as a label — it disappears when the user types.
  • The required attribute prevents form submission if the field is empty and triggers the browser's built-in validation UI. Pair it with a :required CSS rule for visual indication.
  • Use aria-describedby="hint-id" on an input to associate it with a helper text element — screen readers will read the hint after the label. Use aria-invalid="true" when an input fails validation.
  • Group related inputs with <fieldset> and provide a group label with <legend> as its first child. This is especially important for radio buttons and checkboxes — without it, each option lacks context.
  • Style form controls consistently using the same CSS variables as the rest of your page. Use input:valid and input:invalid pseudo-classes to indicate validation state visually, but only after the user has interacted with the field (:user-invalid in modern browsers handles this automatically).
  • Security: always use HTTPS so form data is encrypted in transit. Never trust client-side validation alone — always validate and sanitize inputs on the server. Protect against CSRF with tokens, against XSS by escaping output, and against SQL injection by using parameterized queries.

Reuseable Components

  • Use meaningful, descriptive class names that reflect purpose rather than appearance — e.g., .btn-primary instead of .btn-red. This makes it easier to restyle by changing the CSS variable rather than renaming classes across the HTML.
  • Scope component styles to their own class so they don't bleed into unrelated elements. Avoid targeting bare element types (e.g., button { ... }) in component stylesheets — target the component class instead (.btn { ... }).
  • For buttons: write a base .btn class with shared properties (padding, border-radius, font, transition, cursor), then add modifier classes (.btn-primary, .btn-outline) that only override what differs. This avoids duplicating shared properties across variants.
  • Use CSS custom properties (--variable-name) to centralize design tokens (colors, spacing, border-radius). Component styles then reference the tokens, so a theme change in one place propagates everywhere.

Css Styling Essentials

  • CSS is a set of rules that control the visual presentation of HTML. Each rule has a selector that targets one or more elements, and a block of declarations that define property–value pairs.
  • Each declaration follows the format property: value; — the semicolon is required between declarations. Example: border-radius: 24px;
  • The browser reads stylesheets top to bottom. When two rules target the same element with the same specificity, the one declared later wins. This is the "cascade" in Cascading Style Sheets.
  • Link an external stylesheet in <head> with <link rel="stylesheet" href="css/file.css">. Use a <style> block in <head> for small, page-specific styles. Use inline style="" attributes only for values that must be set at the element level (e.g., dynamic values from JavaScript).
  • There are four main selector types: type selectors (p), class selectors (.card), ID selectors (#header), and attribute selectors ([type="text"]). Classes are preferred for reusable styling; IDs should be used sparingly and mainly for anchor targets or JavaScript hooks.
  • Later rules override earlier ones only when the selectors have equal or higher specificity. A class (.btn-red) overrides a type rule (button) regardless of order.
  • Inline style="" attributes override any external or internal stylesheet declarations for the same property, short of !important.
  • Some CSS properties — like color, font-family, font-size, and line-height — are inherited from parent elements. Others, like padding, margin, border, and display, are not and must be set explicitly.
  • Only use !important as a last resort — it breaks the normal cascade and makes future debugging difficult. Prefer increasing selector specificity instead.
  • Specify colors as hex (#ff2627), RGB (rgb(255, 38, 39)), HSL (hsl(360, 100%, 57%)), or the modern oklch() format. HSL is often easier to adjust by eye — changing the L value lightens or darkens the color.
  • Control typography with font-family, font-size, font-weight (100–900, where 400 is normal and 700 is bold), line-height, letter-spacing, and text-align.
  • The box model: margin controls space outside the element's border, padding controls space between the content and the border, and border sits between them. Set box-sizing: border-box globally so width and height include padding and border.
  • The display property determines layout behavior. Key values: block (full-width, stacks vertically), inline (flows in text), inline-block (flows in text but respects width/height), flex (one-axis flexible layout), grid (two-axis grid layout).

Selectors & Specificity

  • The universal selector * matches every element. Commonly used in resets: *, *::before, *::after { box-sizing: border-box; }. High usage can have minor performance costs, but in practice this is rarely a real concern.
  • Selector types (in order of increasing specificity): type (p), class (.card), attribute ([href]), pseudo-class (:hover), ID (#main), inline style, !important.
  • Combinators: .card p — descendant (any p inside .card); .card > h3 — direct child only; h2 + p — adjacent sibling (immediately after); h2 ~ p — general sibling (any p after an h2 in the same parent).
  • Pseudo-classes target elements in a particular state: :hover, :focus, :focus-visible, :active, :visited, :checked, :disabled, :nth-child(n), :first-child, :last-child, :not(selector). Pseudo-elements target a part of an element: ::before, ::after, ::placeholder, ::first-line.
  • Specificity is calculated as a score of three parts: (ID count, class/attribute/pseudo-class count, element/pseudo-element count). For example, #nav .btn:hover scores (1,2,0). Higher score wins regardless of source order. Inline styles score (1,0,0,0) and beat everything except !important.
  • Write selectors from least to most specific within each stylesheet layer. Overly specific selectors are hard to override cleanly and make the codebase fragile.
  • Use browser DevTools (Inspect Element → Styles panel) to see exactly which rules are applied, which are overridden (shown with strikethrough), and where each rule is declared.

    • Cascade & Inheritance

    • The cascade resolves conflicts between rules in three steps, in priority order: (1) Origin and importance — browser defaults lose to author styles; !important declarations win over normal ones from the same origin. (2) Specificity — higher specificity score wins. (3) Source order — when specificity is equal, the later declaration wins.
    • CSS layers (@layer) are a modern tool for managing cascade origin order explicitly, useful in large codebases to prevent specificity wars.
    • Inheritance: some properties like color, font-family, font-size, line-height, and visibility are automatically inherited from parent elements. Others like padding, margin, border, background, and display are not and start at their initial value.
    • Use inherit to force a non-inheriting property to take its parent's computed value. Use initial to reset a property to its CSS specification default. Use revert to reset to the browser's user-agent stylesheet value (e.g., making a <button> look like a browser default button again).
    • Specificity is calculated as (inline, ID, class/attribute/pseudo-class, element/pseudo-element). Example: #nav .btn:hover a = (0, 1, 2, 1).
    • !important overrides normal cascade for that property. Use it only to fix genuine conflicts you cannot resolve by adjusting specificity. Remove it once the underlying rule is corrected. Stacking !important declarations creates compounding cascade problems.
    • Use DevTools' Styles panel to trace exactly which rule is applying a value, why others are overridden, and what computed value the browser is using.

    HTML & CSS Typography

    • Typography controls how text is rendered — including font choice, size, weight, spacing, and alignment — and has a major effect on readability and perceived quality.
    • System font stacks use fonts already on the user's device, resulting in zero download cost and native-feeling text: font-family: system-ui, -apple-system, sans-serif;. Web-hosted fonts like Google Fonts give you specific typefaces but add a network request.
    • Set a base font on body with a fallback stack: font-family: 'Your Font', system-ui, sans-serif;. All elements inherit this unless overridden. Limit the page to 2–3 font families to keep the design cohesive.
    • Use relative units for font sizes: rem scales relative to the root (html) font size (typically 16px by default), so it respects the user's browser font preferences. em scales relative to the element's own parent — useful for components that should scale with their context, but can compound unexpectedly when nested.
    • font-weight accepts numeric values (100–900) or keywords (normal = 400, bold = 700). Only weights supported by the loaded font render correctly; unsupported weights are synthesized by the browser. font-style: italic renders text in italic.
    • line-height controls the vertical space between lines of text. A unitless value like 1.5 is preferred over px or em because it scales proportionally with the element's font size. letter-spacing adjusts space between characters.
    • text-align controls horizontal alignment (left, right, center, justify). text-transform changes case (uppercase, lowercase, capitalize). text-decoration adds or removes underlines; you can control its color and thickness separately with text-decoration-color and text-decoration-thickness.
    • Use clamp(min, preferred, max) to create fluid font sizes that scale with viewport width without media queries. Example: font-size: clamp(1rem, 2.5vw, 2rem);
    • Normalize type defaults on base elements — set body, h1–h6, p, and a explicitly rather than relying on browser defaults, which vary across browsers.
    • WCAG AA requires a minimum contrast ratio of 4.5:1 for normal text and 3:1 for large text (18pt+ or 14pt bold). Don't lock absolute font sizes — allow the user's browser zoom and font preference settings to work. Use semantic elements (<strong>, <em>) for meaning rather than just visual styling.

    Colors & Backgrounds

    • Color values can be specified as hex (#ff2627), rgb(255, 38, 39), hsl(360, 100%, 57%), or the modern oklch() format. HSL is often easiest to adjust — change L (lightness) to create tints and shades. oklch() is perceptually uniform, meaning equal changes in L always look equally different.
    • Define colors as CSS custom properties (variables) at the :root level: --primary-red: #ff2627;. Reference them with var(--primary-red). This makes global color changes a one-line edit and enables theming.
    • background-color fills the element with a solid color. background-image sets an image or gradient. Both can be layered by separating values with commas (images render on top). background-size: cover scales the image to fill the container; background-position: center centers it; background-repeat: no-repeat prevents tiling.
    • CSS gradients: linear-gradient(direction, color1, color2) transitions colors in a straight line; radial-gradient(shape, color1, color2) radiates outward from a center point; conic-gradient(color1, color2) sweeps around a center point like a pie chart. Gradients are image values, so they go in background-image, not background-color.
    • To overlay a gradient on a background image, comma-separate them in background-image: background-image: linear-gradient(...), url(image.jpg); — the gradient is rendered on top.
    • Data URIs and CSS gradients can create subtle textures and patterns without additional image files, keeping asset requests low.
    • Ensure all text meets WCAG AA contrast requirements: 4.5:1 for normal text, 3:1 for large text (18pt+). Use a contrast checker tool to verify color pairs before shipping.

    Box Model

    • Every HTML element is rendered as a rectangular box composed of four layers from inside out: content → padding → border → margin.
    • Content is the area where text and child elements appear. Padding is transparent space between the content and the border. Border wraps the padding and content. Margin is transparent space outside the border that separates the element from its neighbors.
    • By default, width and height apply only to the content area. With box-sizing: border-box;, they instead apply to the total box including padding and border. Apply this globally: *, *::before, *::after { box-sizing: border-box; } — otherwise width calculations are unintuitive.
    • DevTools' box model diagram (Inspect Element → Computed tab) shows the exact pixel values for each layer of the selected element.
    • Margin collapsing: when two block elements are adjacent vertically, their margins collapse to the larger of the two values rather than adding together. A parent element and its first or last child can also share margins unless the parent has padding, a border, or triggers a block formatting context.
    • To prevent unexpected margin collapse between a parent and its first child, add padding-top: 1px or a border to the parent, or give the parent overflow: hidden or display: flow-root.
    • Shorthand: margin: 24px 16px; sets 24px top/bottom and 16px left/right. Four values follow clockwise: top, right, bottom, left (TRBL). The same pattern applies to padding and border-width.
    • Define a spacing scale using custom properties (e.g., --space-sm: 8px; --space-md: 16px; --space-lg: 24px;) and apply them consistently. This keeps margins and padding proportional across the page and makes adjustments easy.

    Layout Fundamentals

    • Normal flow: block-level elements (<div>, <p>, <section>) stack vertically and take the full available width. Inline elements (<span>, <a>, <strong>) flow horizontally with text and ignore width/height. Inline-block elements flow like inline but respect width and height.
    • Change an element's layout behavior with display: value;. Common values: block, inline, inline-block, flex, inline-flex, grid, inline-grid, none (removes from layout entirely).
    • float: left or float: right removes an element from normal flow and allows text to wrap around it. The parent container collapses in height and doesn't contain floated children. Clear a float container with overflow: auto, overflow: hidden, or the clearfix method (::after { content: ""; display: table; clear: both; }).
    • Prefer flexbox for one-dimensional layouts and grid for two-dimensional layouts. Floats are mostly legacy — you'll encounter them in older codebases.
    • Elements render in DOM (source) order by default. Positioning and z-index can change the visual stacking order independently of source order, but source order still determines tab order for keyboard users.
    • A new stacking context is formed by elements with: position + a z-index other than auto; opacity less than 1; transform; filter; isolation: isolate; and several others. Elements inside a stacking context are z-ordered relative to each other, not to the rest of the page. This is a common source of z-index issues.

    Positioning & Z-index

    • Positioning lets you place elements independently of normal document flow. Use it for overlays, tooltips, badges, sticky headers, and fixed UI chrome — not as a general layout tool (use flexbox/grid for that).
    • Position values: static (default, ignores top/left/right/bottom and z-index); relative (stays in normal flow but offsets from its original position using top/left/right/bottom; creates a containing block for absolutely positioned children); absolute (removed from flow, positioned relative to the nearest non-static ancestor); fixed (removed from flow, positioned relative to the viewport and stays there during scroll); sticky (acts as relative until it reaches a scroll threshold, then behaves as fixed within its scroll container).
    • Combine position with top, right, bottom, left, and inset (shorthand for all four) to control placement. These offsets have no effect on static elements.
    • A common pattern: set position: relative on a parent, then position: absolute; top: 0; right: 0; on a child (like a badge or close icon) to anchor it to the parent's corner.
    • z-index only works on elements with a position value other than static, and on flex or grid children. Higher z-index renders on top. Elements in different stacking contexts are compared at the context level, not element by element — this is why a high z-index sometimes appears behind a lower-z-index element.
    • For position: sticky to work: the sticky element must have a top, left, right, or bottom threshold set, and its scroll container must have overflow set to something other than visible (or no overflow set at all). A common bug is a parent with overflow: hidden breaking sticky behavior.
    • Use DevTools to inspect stacking contexts when overlays misbehave — it highlights which element creates each context.

    Flexbox Layout

    • Flexbox is a one-dimensional layout model — it arranges items along either a row or a column and handles distribution of space and alignment along those axes. Ideal for navigation bars, card rows, form layouts, and centering.
    • Enable flexbox on a container with display: flex; (block-level) or display: inline-flex; (inline-level). The immediate children become flex items automatically.
    • flex-direction: row (default) lays items left-to-right; row-reverse reverses that. column stacks vertically; column-reverse reverses it. This sets the main axis.
    • The main axis follows flex-direction. Control spacing along it with justify-content: flex-start, flex-end, center, space-between, space-around, space-evenly. The cross axis is perpendicular — align items along it with align-items on the container or align-self on individual items.
    • Use gap: row-gap column-gap; for consistent spacing between items. This is preferred over margins because it only adds space between items, not on the outer edges.
    • flex shorthand on items: flex: grow shrink basis;. flex-grow defines how much of remaining space the item takes (0 = don't grow). flex-shrink defines how the item shrinks when space is tight (0 = don't shrink). flex-basis sets the starting size before space is distributed. Common shorthands: flex: 1; (equal width, same as 1 1 0%), flex: 0 0 300px; (fixed 300px, doesn't grow or shrink).
    • flex-wrap: wrap; lets items wrap onto new rows or columns when they exceed the container width. Without it, items shrink to fit or overflow. align-content aligns the rows/columns when wrapping occurs.
    • order changes the visual rendering order of flex items without altering DOM order. Note: DOM order still controls tab order, so changing visual order with order can create confusing keyboard navigation if overused.

    Grid Layout

    • CSS Grid is a two-dimensional layout system — it handles rows and columns simultaneously. Best for page-level layouts, dashboards, image galleries, and any structure that needs alignment in both axes.
    • Enable with display: grid; (block-level) or display: inline-grid;. Immediate children become grid items.
    • Define columns with grid-template-columns and rows with grid-template-rows. Use px for fixed tracks, fr (fractional unit) to distribute remaining space, auto to size to content, or minmax(min, max) for flexible bounds.
    • gap (or row-gap / column-gap) sets track spacing. The old grid-gap property is deprecated; use gap.
    • Fixed-width first column, equal remaining: grid-template-columns: 300px 1fr 1fr;. The 1fr units share the space left after the 300px column.
    • Responsive fluid grid without media queries: grid-template-columns: repeat(auto-fit, minmax(220px, 1fr)); — auto-fit collapses empty tracks so existing items stretch to fill; auto-fill keeps empty tracks. minmax(220px, 1fr) sets a minimum column width of 220px and a maximum of one fractional unit.
    • Grid items auto-place in source order by default. Explicitly place items with grid-column: start / end; and grid-row: start / end; using line numbers. Use span to span multiple tracks: grid-column: 1 / span 3;
    • Named areas: define a visual layout in ASCII with grid-template-areas on the container, then assign items to areas with grid-area: name;. Use . for empty cells.
    • Alignment inside tracks: justify-items (inline axis) and align-items (block axis) on the container. Override per item with justify-self and align-self. Use place-items: center; as shorthand for both axes.
    • Alignment of the whole grid within its container: justify-content and align-content, or place-content shorthand. Useful when the total track size is smaller than the container.
    • For responsive breakpoints, adjust grid-template-columns and grid-template-areas inside media queries to reflow the layout at different screen widths.

    Layout Patterns

    Combine Flexbox, Grid, and modern CSS features to build reusable page layouts — hero sections, split panels, dashboards, and marketing pages — without resorting to brittle floats.

    General Principles

    • Use flexbox and grid as your primary layout tools. Avoid floats for new layouts — they exist in legacy code but are not the right tool for modern page structure.
    • Centering: display: grid; place-items: center; or display: flex; justify-content: center; align-items: center; on a container are the cleanest ways to center a child both horizontally and vertically.

    Hero with Split Emphasis

    Pair messaging with media using a two-column grid that collapses on smaller screens. "Collapsing" means the columns stack vertically when the screen is narrow.

    • Use grid-template-columns: repeat(auto-fit, minmax(280px, 1fr)); so the two columns stack automatically on small screens — no media query needed.
    • Use clamp() for fluid padding and font sizes that scale with viewport width without breakpoints: e.g. font-size: clamp(2.5rem, 6vw, 3.75rem);.
    • Apply align-items: center; on the grid container to vertically align the text column and media column.

  gap: 32px;
  align-items: center;
  padding: clamp(40px, 8vw, 120px) clamp(24px, 6vw, 80px);
  background: radial-gradient(circle at top left, #2563eb, #0f172a);

    Split Panel / App Shell

    Use Grid to balance a fixed sidebar against a scrollable content area. An "application shell" is the basic frame of the app — sidebar, header, and main area — without page-specific data.

    • Define two columns with grid-template-columns: 280px 1fr; on the outer container and set min-height: 100vh; so it fills the viewport.
    • At a mobile breakpoint, collapse to grid-template-columns: 1fr; and make the nav position: sticky; top: 0; so it stays visible while the user scrolls the main content.
    • Use grid-template-areas to name the regions and reassign them in media queries for mobile layouts.

    Dashboard / Responsive Card Grid

    Create responsive card grids that adapt gracefully from desktop to mobile without any media queries.

    • Use grid-template-columns: repeat(auto-fit, minmax(260px, 1fr)); — auto-fit collapses empty tracks so existing items stretch to fill, while minmax() sets the minimum card width.
    • Use auto-fill instead of auto-fit if you want empty placeholder tracks to remain (useful for keeping a consistent column count).
    • For highlighted "featured" cards, use grid-template-areas or grid-column: span 2; to make specific cards span multiple columns.
    • Combine clamp() for adaptive padding and typography so card content scales with the viewport.

    Marketing Content Blocks (Alternating Text & Image)

    Alternate text and imagery using a reusable flip class or :nth-child selectors to create visual rhythm without repeating code.

    • Wrap each section in a two-column grid: grid-template-columns: 1fr 1fr;.
    • Flip the column order on alternate rows via the order property (Flexbox) or by redefining grid-template-areas on a .flip class.
    • Stack vertically under a mobile breakpoint by collapsing to grid-template-columns: 1fr; and removing the order override.

    Footer Patterns

    Combine CSS Grid for multi-column link groups and Flexbox for legal copy alignment.

    • Use grid-template-columns: repeat(auto-fit, minmax(160px, 1fr)); for link clusters so columns adapt to the available width automatically.
    • Place social icons in a Flexbox row: display: flex; gap: 16px; align-items: center;.
    • Separate the main link grid from the bottom legal/copyright bar with a border or extra margin, and use justify-content: space-between; on the legal row to push copyright text left and policy links right.

    Responsive Design

    Responsive design ensures every layout looks and works great on any device. Combine flexible grid systems, fluid typography, and media queries to adapt content gracefully.

    Viewport Basics

    • Always include the viewport meta tag: <meta name="viewport" content="width=device-width, initial-scale=1">. Without it, mobile browsers render at a desktop width and scale everything down, breaking your layout.
    • Design mobile-first — write your base styles for narrow screens, then layer on media queries to enhance for tablets and desktops. This approach keeps CSS leaner and forces you to prioritise essential content.
    • Use relative units (rem, vw, %) rather than px wherever possible so elements scale naturally with the viewport and user font-size preferences.

    Fluid Grids

    • Use Flexbox and Grid with fractional units (fr) to create layouts that flow and resize rather than overflow. Fractional units divide up leftover space evenly between tracks.
    • Apply max-width to text containers (e.g. max-width: 65ch;) to prevent line lengths from becoming unreadably wide on large screens.
    • Use repeat(auto-fit, minmax(min, 1fr)) on grid containers for automatic, breakpoint-free column counts — columns form when there's room and collapse to one when there isn't.

    Responsive Typography

    • Use clamp(min, preferred, max) for font sizes: e.g. font-size: clamp(1rem, 2vw, 1.5rem);. The browser picks a value between the minimum and maximum, keeping text readable at every viewport width.
    • Scale headings, line heights, and spacing together so the vertical rhythm holds at all sizes. If the heading grows, the surrounding white space should grow proportionally.
    • Consider defining fluid spacing tokens with CSS custom properties — e.g. --space-md: clamp(1rem, 3vw, 2rem); — so you can update spacing site-wide from one place.

    Media Queries

    • A breakpoint is the screen width at which your layout needs to change. Choose breakpoints based on where the design breaks, not around specific device sizes, since device sizes change constantly.
    • Write mobile-first: base styles apply to all screens; @media (min-width: ...) queries progressively enhance for larger viewports. This is typically more maintainable than overriding desktop styles downward.
    • Common starting points: min-width: 480px (large phones), min-width: 768px (tablets), min-width: 1200px (wide desktops) — but always adjust to suit your content.
    :root {
  --space: clamp(1rem, 2vw, 1.5rem);
}
.page {
  display: grid;
  gap: var(--space);
}
@media (min-width: 768px) {
  .page { grid-template-columns: 240px 1fr; }
}
@media (min-width: 1200px) {
  .page { grid-template-columns: 280px 2fr 1fr; }
}

    Container Queries

    • Standard media queries react to the viewport width; container queries (@container) react to the width of a parent element. This makes components truly self-contained and reusable in different layout contexts.
    • Opt a parent into container query tracking with container-type: inline-size;. Then use @container (min-width: 400px) { ... } inside child selectors.
    • Browser support is now good (all modern browsers), but still provide sensible default styles as a fallback for older environments.

    Responsive Media

    • Make all images fluid by default: img { max-width: 100%; height: auto; }. This prevents images from overflowing their containers while maintaining their aspect ratio.
    • Use the <picture> element with <source media="..."> attributes to serve different image files at different breakpoints — useful for art direction (e.g. a cropped portrait on mobile, a wide landscape on desktop).
    • For embedded videos and iframes, preserve aspect ratio with aspect-ratio: 16 / 9; combined with width: 100%;, or use the older padding-percentage hack for broader support.
    • Use object-fit: cover; on <img> elements inside fixed-size containers to crop cleanly without distorting the image.

    Testing Strategy

    • Use browser DevTools device mode to quickly preview common breakpoints and simulate touch events. Resize the viewport manually to catch edge cases between presets.
    • Test on real devices when possible — DevTools emulation doesn't fully replicate touch target size, font rendering, or scrolling behaviour on actual hardware.
    • Check landscape and portrait orientations, and test with the browser's minimum and maximum font-size settings to ensure your fluid type still holds.
    • Audit performance: responsive sites should serve appropriately sized images and avoid loading desktop-weight resources on mobile. Use the Network tab in DevTools to check payload sizes.

    Responsive Navigation

    Responsive Cards

    Cards should stack in a single column on narrow screens and form a multi-column grid on wider ones, without needing explicit breakpoints for every step.

    • Use grid-template-columns: repeat(auto-fit, minmax(260px, 1fr)); for fluid card grids — columns form automatically when there is enough room and collapse to one when there isn't.
    • Keep a consistent content hierarchy inside every card: image → heading → supporting text → call-to-action button. Changing this order between cards creates visual noise and makes scanning harder.
    • Give cards a fixed min-height or use display: flex; flex-direction: column; with margin-top: auto; on the CTA button so all cards in a row share the same height and their buttons line up.
    • Prefer aspect-ratio on card images rather than fixed pixel heights, so images scale correctly at every card width without cropping or distorting unexpectedly.

    Data Tables

    Wide tables overflow on small screens. Rather than forcing horizontal scrolling, consider restructuring the data for mobile.

    • The simplest approach is overflow-x: auto; on a wrapper element — this contains the overflow and lets the user scroll the table horizontally without the rest of the page scrolling.
    • For a richer experience, use display: block; on tr, td, and th at narrow widths to transform rows into stacked card-like blocks. Use data-label attributes and CSS ::before pseudo-elements to re-inject column headings next to each value.
    • Hide non-essential columns at narrow widths with display: none; inside a media query. Prioritise the columns a mobile user actually needs.
    • Whatever layout you choose, always preserve the semantic <table>, <thead>, and <th scope="..."> structure so screen readers can still announce the relationship between each cell and its header.

    Responsive Forms

    Forms need to be easy to complete on both a desktop keyboard and a phone touchscreen.

    • On desktop, arrange label-and-input pairs in a two-column grid (grid-template-columns: auto 1fr;) so labels and fields align neatly. On mobile, collapse to a single column with grid-template-columns: 1fr; so labels sit above their inputs.
    • Always link every <label> to its input via matching for and id attributes. Never rely solely on placeholder text as a label — placeholders disappear as soon as the user starts typing.
    • Set width: 100%; on inputs and buttons at narrow viewports so they become easy tap targets that fill the available width.
    • Use the correct type attribute on inputs (type="email", type="tel", type="number") so mobile browsers display the appropriate keyboard automatically.

    Micro-interactions & Touch Considerations

    Small animations and state changes make interfaces feel responsive to input — but they must work across pointer, keyboard, and touch.

    • Never style only :hover. Always pair it with :focus-visible so keyboard users get the same visual feedback. On touch devices, :hover can be sticky or entirely absent.
    • Wrap animations in @media (prefers-reduced-motion: no-preference) { ... } so they only run for users who haven't requested reduced motion at the OS level. Users with vestibular disorders can experience nausea from excessive motion.
    • All interactive elements — buttons, links, form controls — should have a minimum tap target size of 44×44 px as recommended by WCAG. Use min-height and padding rather than setting a fixed height, so the element can still grow with its content.
    • Use CSS transition on transform and opacity for animations rather than animating width, height, or margin — transform and opacity changes are handled by the compositor thread and don't cause layout recalculations, so they stay smooth even on low-powered devices.

    CSS Custom Properties (Variables)

    • CSS custom properties — commonly called variables — are defined with a -- prefix and read back with the var() function. They are case-sensitive: --Color-Brand and --color-brand are distinct properties.
    • Declare global tokens on :root, which targets the <html> element and gives them the highest possible scope. Any element in the document can then inherit and consume them via var(--token-name).
    • Custom properties are part of the cascade and inherit just like color or font-size. Redefining a variable on a container element — e.g. .card { --surface: #1e293b; } — scopes the new value to that element and all of its descendants, leaving the global definition untouched.
    • Provide a fallback value as the second argument to var(): var(--space-md, 1rem). The fallback is used whenever the referenced variable has not been defined in the current scope, acting as a safe default rather than causing an invalid value.
    • Variables work seamlessly inside calc(), clamp(), and other CSS functions: padding: calc(var(--space-md) * 1.5); or font-size: clamp(var(--text-min), 3vw, var(--text-max));. The variable is substituted first, then the function evaluates.
    • Switch themes at runtime by toggling a data-theme attribute on <html> and redefining the same token names under that selector. Because all components reference the same variable names, every colour, spacing value, and shadow updates automatically with a single attribute change — no JavaScript needs to touch individual elements.
    • The @media (prefers-color-scheme: dark) query can redefine custom properties directly, giving users an automatic dark theme that respects their OS preference without any JavaScript at all.
    • JavaScript can read a variable with getComputedStyle(el).getPropertyValue('--token-name').trim() and write one with el.style.setProperty('--token-name', value). Writing to document.documentElement updates the global scope; writing to a specific element scopes the change to that element only.
    • Use @supports (--custom: property) to detect custom property support before applying variable-dependent rules. All modern browsers support custom properties; this guard is only necessary when targeting very old browsers such as IE 11, which has no support at all.
    /* 1. Define global design tokens on :root */
:root {
  --color-brand:    #2563eb;
  --color-surface:  #ffffff;
  --color-text:     #0f172a;
  --space-md:       clamp(1rem, 2vw, 1.5rem);
  --radius-card:    12px;
*}

/* 2. Dark theme — same names, different values */
:root[data-theme="dark"] {
  --color-surface: #0f172a;
  --color-text:    #e2e8f0;
}

/* 3. Consume tokens anywhere */
body {
  background: var(--color-surface);
  color:      var(--color-text);
  transition: background 0.3s ease, color 0.3s ease;
}

.card {
  padding:       var(--space-md);
  border-radius: var(--radius-card);
  background:    var(--color-surface);
}

.btn-primary {
  background: var(--color-brand);
  color: #fff;
  padding: calc(var(--space-md) * 0.5) var(--space-md);
}

/* 4. Scope a token override to a single component */

  --color-surface: #1e293b; /* only affects this card and its children */
}

    CSS Functions & Calculations

    CSS functions let you compute values, blend colors, and respond to context without extra JavaScript. Combine them with custom properties to build smart, responsive design systems.

    Math Helpers

    • calc() — performs arithmetic across mixed units directly in CSS. Example: width: calc(100% - 2rem); subtracts a fixed rem value from a fluid percentage — something that is impossible with a plain static value. Spaces around the operator are required by the spec.
    • min() / max() — pick the smallest or largest value from a comma-separated list. Example: width: min(600px, 100%); keeps an element at most 600px wide but never overflows its container. Multiple values and mixed units are supported.
    • clamp(min, preferred, max) — constrains a value to a responsive range. The middle (preferred) value is typically a viewport unit so it scales fluidly; the first and third arguments act as hard floor and ceiling. Example: font-size: clamp(1rem, 2.5vw, 1.5rem); scales with the viewport but never smaller than 1rem or larger than 1.5rem.
    /* Fluid spacing scale using clamp() — scales between breakpoints
   without any media queries. Custom properties make the scale
   reusable across the entire project. */
:root {
  --space-xs: clamp(0.5rem,  1vw,   0.75rem);
  --space-sm: clamp(0.75rem, 1.5vw, 1.125rem);
  --space-lg: clamp(1.5rem,  3vw,   2.5rem);
}

.card {
  padding:       var(--space-lg);
  margin-block:  var(--space-sm);
  /* calc() uses the custom property as an operand */
  border-radius: calc(var(--space-xs) * 2);
}

/* width can never exceed 800px, but is otherwise fluid */
.content-area {
  width: min(800px, 100% - var(--space-lg) * 2);
}

    Color Functions

    • rgb() and hsl() accept decimal values and an optional alpha channel (0 = fully transparent, 1 = fully opaque). Modern syntax omits the slash: rgb(99 102 241 / 0.5). Prefer hsl() when you need predictable lightness adjustments — the third argument maps directly to perceived brightness.
    • color-mix(in srgb, colorA 40%, colorB) blends two colors at a defined ratio in a given color space. This removes the need for hardcoded intermediate swatches and makes theme variations easier to maintain — change one source color and every blend updates automatically.
    • color-contrast() is still experimental — it returns whichever of a supplied list provides the best contrast ratio against a base color. Where supported, it can automate accessible text-on-background decisions.
    :root {
  --brand: hsl(231 89% 63%);          /* indigo */
  --brand-muted: color-mix(in srgb, var(--brand) 40%, white);
}

/* Semi-transparent overlay using modern rgb() syntax */
.overlay {
  background: rgb(0 0 0 / 0.45);
}

/* Automatic light/dark text based on background */
.badge {
  background: var(--brand);
  color: color-contrast(var(--brand) vs white, black);
}

    Trigonometry & Geometry

    • Modern browsers support sin(), cos(), tan(), asin(), acos(), atan(), atan2(), and hypot() natively in CSS. Arguments are in radians by default; use deg units for degrees.
    • These are most useful for polar-coordinate animations, radial menu positioning, or wave-shaped decorative elements — all achievable without JavaScript or SVG hacks.
    • Pair with CSS custom properties and @property to animate the angle value, producing smooth orbital or carousel motion driven entirely by CSS transitions.
    /* Position items in a circle using sin/cos.
   --angle is incremented per item with calc(). */
.orbit-item {
  --angle: 45deg;
  --radius: 120px;
  position: absolute;
  top:  calc(50% + var(--radius) * sin(var(--angle)));
  left: calc(50% + var(--radius) * cos(var(--angle)));
}

    Environment & Context Helpers

    • env(safe-area-inset-bottom) (and top/left/right) exposes the safe-area insets that mobile browsers report for notches and home bars. Always add these to padding on fixed footers and sticky headers when targeting mobile web apps.
    • var(--name, fallback) retrieves a custom property value; the optional second argument is used whenever the variable is undefined in the current scope. This makes var() a context helper as much as a variable lookup — components can declare their own local fallbacks and stay self-contained.
    • attr(data-label) reads an HTML attribute inside a CSS content declaration. It is commonly used to surface data-label values on td elements when a data table is reflowed into a card layout at narrow viewports.
    /* Fixed footer that clears the iOS home-bar safe area */

  position: sticky;
  bottom: 0;
  padding-bottom: calc(1rem + env(safe-area-inset-bottom));
}

/* Responsive table: surface column header as a pseudo-label */
@media (max-width: 600px) {
  td::before {
    content: attr(data-label) ": ";
    font-weight: 700;
  }
}

    CSS Transitions

    CSS transitions animate changes to CSS properties, creating smooth hover states, focus styles, and UI feedback without JavaScript. They activate automatically whenever a targeted property changes value — no scripting required.

    Core Properties

    • transition-property — names the CSS property (or properties) to animate, e.g. background-color, transform, opacity. Use all to target every animatable property at once, but avoid it in production: it causes the browser to watch every property for changes, which is wasteful and can produce unintended animations on properties you never meant to transition.
    • transition-duration — total running time of the animation, written as a time value (200ms, 0.3s). There is no default — a missing or zero duration means the change is instant.
    • transition-timing-function — the easing curve that controls acceleration through the animation. Common keywords are ease (default — slow start, fast middle, slow end), linear, ease-in (starts slow), ease-out (ends slow), and ease-in-out. Custom curves use cubic-bezier().
    • transition-delay — how long the browser waits before starting the animation after the property change is detected. Useful for staggering a sequence of elements.
    • transition shorthand — sets all four in one declaration in the order property duration timing-function delay. Example: transition: background 180ms ease, transform 180ms ease;. Multiple transitions are comma-separated, each targeting a different property.
    /* Shorthand: property | duration | timing-function | delay
   Two properties transitioned independently in one rule. */
.cta {
  display: inline-flex;
  align-items: center;
  gap: 12px;
  padding: 12px 22px;
  border-radius: 999px;
  background: #2563eb;
  color: #fff;
  text-decoration: none;
  /* background and transform each get their own curve */
  transition: background 180ms ease, transform 180ms ease;
}

/* :focus-visible ensures keyboard users see the same feedback
   as pointer users — never style :hover alone */
.cta:hover,
.cta:focus-visible {
  background: #1d4ed8;   /* darker blue */
  transform: translateY(-2px); /* subtle lift */
}

    Timing Functions

    • ease — the default; accelerates quickly then decelerates. Works well for most UI interactions because it feels natural to the human eye.
    • linear — constant speed throughout. Best for looping animations (spinners, progress bars) where you want no perceived acceleration.
    • ease-in — starts slow, ends fast. Use sparingly for elements leaving the screen (exits), where the acceleration feels intentional.
    • ease-out — starts fast, ends slow. Ideal for elements entering the screen (entrances) — the deceleration implies the element has arrived and settled.
    • ease-in-out — slow at both ends, fast in the middle. Good for longer transitions where you want a smooth, polished feel throughout.
    • cubic-bezier(x1, y1, x2, y2) — define any custom curve using two control points. Tools like cubic-bezier.com let you preview curves visually before committing them to code. Values can exceed 0–1 on the Y axis to create overshoot (bounce) effects.
    • steps(n, direction) — divides the transition into n equal discrete jumps instead of a smooth curve. Use it for sprite-sheet animations, typewriter effects, or any scenario where intermediate states should not be visible.
    /* Custom spring-like overshoot using cubic-bezier.
   Y values above 1.0 cause the element to "bounce" past
   the target before settling — impossible with keyword easings. */
.spring {
  transition: transform 400ms cubic-bezier(0.34, 1.56, 0.64, 1);
}
.spring:hover {
  transform: scale(1.08);
}

/* steps() — jumps between frames, no interpolation.
   Perfect for a typewriter cursor blink. */
.cursor {
  animation: blink 1s steps(1, end) infinite;
}
@keyframes blink {
  50% { opacity: 0; }
}

    Responsive Motion & Accessibility

    Hover states and motion effects do not exist on touch screens or for users who have requested reduced motion at the operating system level. Any interaction that relies solely on :hover or on animation must have a respectful fallback.

    • Wrap all non-essential motion inside @media (prefers-reduced-motion: no-preference) { ... } so transitions only run when the user has not requested reduced motion. The inverse query (prefers-reduced-motion: reduce) can be used to explicitly supply a stripped-down fallback with no movement. This is an operating-system accessibility setting; honoring it prevents discomfort or harm for users with vestibular disorders.
    • Keep interactive feedback durations between 150ms and 300ms. Shorter feels snappy and responsive; beyond 300ms starts to feel sluggish. Reserve longer durations (400–600ms) only for complex transitions like page-level layout changes.
    • Avoid large transform distances or significant layout-shifting transitions — a button that moves 80px on hover is jarring; 2–4px is the practical sweet spot for a lift effect.
    • Always pair :hover styles with :focus-visible so keyboard users receive identical visual feedback. :focus-visible (rather than :focus) shows the indicator only when navigating by keyboard, avoiding the focus ring on mouse clicks.
    • Do not transition width, height, top, or left — these trigger full layout recalculations on every frame. Prefer transform: scale() and transform: translate(), which are composited by the GPU and do not affect document flow.
    /* Safe default: no motion at all */
.btn {
  background: #2563eb;
  color: #fff;
}
.btn:hover,
.btn:focus-visible {
  background: #1d4ed8;
}

/* Motion layer: only applied when the user has NOT
   requested reduced motion at the OS level */
@media (prefers-reduced-motion: no-preference) {
  .btn {
    transition: background 180ms ease, transform 180ms ease;
  }
  .btn:hover,
  .btn:focus-visible {
    transform: translateY(-2px);
  }
}

/* GPU-friendly: use transform instead of top/left/width/height */
.panel {
  /* BAD  — triggers layout: width: 200px → width: 400px  */
  /* GOOD — composited, no reflow:                         */
  transform: scaleX(0.5);
  transform-origin: left center;
  transition: transform 250ms ease-out;
}
.panel.open {
  transform: scaleX(1);
}

    2D Transforms

    • translate(x, y) — moves an element along the X and/or Y axis without disturbing surrounding elements. Positive X moves right; positive Y moves down. Single-axis shorthand: translateX() / translateY(). Accepts any CSS length unit or percentage; percentages are relative to the element's own dimensions, making translate(-50%, -50%) a reliable centring technique.
    • scale(x, y) — resizes an element around its transform origin. A value of 1 is unchanged, 2 doubles the size, 0.5 halves it. If only one argument is given, both axes scale equally. Use scaleX() or scaleY() to stretch or compress on a single axis.
    • rotate(angle) — spins the element clockwise for positive values, counter-clockwise for negative. Accepts deg, rad, grad, or turn units. Example: rotate(0.25turn) and rotate(90deg) are equivalent.
    • skew(x-angle, y-angle) — slants the element along each axis. Use sparingly — skew distorts text and can reduce readability. Single-axis shorthand: skewX() / skewY().
    • Multiple transforms can be chained in a single transform declaration, separated by spaces: transform: translateY(-4px) scale(1.02) rotate(2deg);. The browser applies them right-to-left, so order matters — rotate then translate produces a different result than translate then rotate.
    /* Chain multiple 2D transforms — order is right-to-left.
   This scales first, then lifts the already-scaled element. */
.card:hover,
.card:focus-visible {
  transform: translateY(-6px) scale(1.02);
  transition: transform 220ms ease;
}

/* Percentage translate: centres absolutely-positioned elements
   regardless of their dimensions */
.modal {
  position: absolute;
  top: 50%;
  left: 50%;
  transform: translate(-50%, -50%); /* pulls back by half own width/height */
}

    3D Transforms

    • perspective(value) — applied directly on the transform property of the element being transformed (inline perspective). Smaller values produce a more dramatic depth effect; larger values flatten it. Example: transform: perspective(700px) rotateY(20deg);. Alternatively, set perspective: 800px; on the parent container so all child elements share the same vanishing point, which creates a more cohesive scene.
    • rotateX(angle) / rotateY(angle) / rotateZ(angle) — rotate around each 3D axis. rotateX tips the top toward or away from the viewer; rotateY swings left and right; rotateZ is equivalent to the 2D rotate().
    • translateZ(value) — moves the element toward or away from the viewer along the Z axis. Positive values bring it closer; negative values push it back. Requires perspective to be visible.
    • transform-style: preserve-3d; — set on a parent to ensure its children are positioned in 3D space rather than flattened into the parent's plane. Without it, nested 3D transforms have no effect on descendants.
    • backface-visibility: hidden; — hides the reverse face of an element when it is rotated more than 90°. Essential for card-flip effects to prevent the back face showing through while the front is visible.
    /* 3D card tilt on hover — perspective set inline so each card
   has its own vanishing point */
.grid {
  display: grid;
  gap: 24px;
  grid-template-columns: repeat(auto-fit, minmax(220px, 1fr));
  padding: clamp(24px, 6vw, 80px);
}

.card {
  padding: 32px;
  border-radius: 24px;
  background: linear-gradient(145deg, #1e293b, #0f172a);
  color: #fff;
  /* baseline: flat, no tilt */
  transform: perspective(700px) rotateX(0deg) rotateY(0deg);
  transition: transform 220ms ease, box-shadow 220ms ease;
}

.card:hover,
.card:focus-visible {
  transform: perspective(700px) rotateX(6deg) rotateY(-6deg) translateY(-6px);
  box-shadow: 0 24px 40px rgb(15 23 42 / 0.35);
}

/* Card-flip pattern — requires preserve-3d on the container */
.flip-container {
  perspective: 1000px;
}
.flip-card {
  transform-style: preserve-3d;
  transition: transform 500ms ease;
}
.flip-card.flipped {
  transform: rotateY(180deg);
}
.flip-card .front,
.flip-card .back {
  backface-visibility: hidden;
  position: absolute;
  inset: 0;
}
.flip-card .back {
  transform: rotateY(180deg);
}

    Transform Origin

    • By default, every transform is applied relative to the element's centre (50% 50%). transform-origin shifts that anchor point, changing how rotation, scaling, and skewing behave.
    • Accepts keyword pairs (top left, bottom right), length values (0 0), or percentages (100% 0%). A third value sets the Z-axis origin for 3D transforms.
    • Common patterns: transform-origin: top center; for a dropdown unfolding from its trigger; transform-origin: left center; for a sliding drawer that grows rightward from its attachment point; transform-origin: bottom center; for a tooltip that scales up from its pointer.
    /* Dropdown menu expands downward from the trigger edge,
   not from its own centre */
.dropdown {
  transform-origin: top center;
  transform: scaleY(0);
  transition: transform 200ms ease-out;
}
.dropdown.open {
  transform: scaleY(1);
}

/* Tooltip scales from its tail point at the bottom */
.tooltip {
  transform-origin: bottom center;
  transform: scale(0);
  transition: transform 150ms cubic-bezier(0.34, 1.56, 0.64, 1);
}
.tooltip.visible {
  transform: scale(1);
}

    Performance Tips

    • Transforms are GPU-accelerated — the browser hands them to the compositor thread and animates them without triggering layout or paint recalculations. This makes transform: translate() dramatically cheaper than changing top, left, margin, or width.
    • will-change: transform; is a hint to the browser that this element is about to be transformed, allowing it to promote the element to its own compositor layer in advance. Use it sparingly and only when you have measured a real performance problem — overusing it consumes GPU memory and can actually slow pages down.
    • Animating opacity alongside transform is safe; both are composited properties. Avoid animating filter, clip-path, or border-radius simultaneously with transforms on low-powered devices — these may fall back to software rendering.
    • Always provide a @media (prefers-reduced-motion: reduce) fallback that removes or simplifies large transform movements. A 180° flip or a 100px translateY that is dramatic on desktop can cause discomfort for users with vestibular disorders.
    /* Default: instant state change, no motion */
.panel {
  transform: translateX(-100%);
}
.panel.open {
  transform: translateX(0);
}

/* Motion enhancement — only added when the user hasn't
   requested reduced motion at the OS level */
@media (prefers-reduced-motion: no-preference) {
  .panel {
    transition: transform 300ms ease-out;
    /* hint to compositor — only add if you've measured a
       real jank problem; remove after profiling */
    will-change: transform;
  }
}
